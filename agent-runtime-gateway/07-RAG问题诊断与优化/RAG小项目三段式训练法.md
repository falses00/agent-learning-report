# RAG 小项目三段式训练法

生成日期：2026-06-30  
用途：把 GitHub 上个人或小团队的 RAG 优化项目转化成学习能力，而不是盲目复制依赖。

## 1. 为什么要看小项目

大框架适合学系统边界，小项目适合学单点优化：

- late chunking 解决“切块后上下文断裂”。
- HyDE/multi-query 解决“查询太短或表达不同”。
- CRAG/Self-RAG 解决“检索失败后如何自修复或拒答”。
- RAPTOR/GraphRAG 解决“多跳、层级总结、跨文档关系”。
- semantic cache 解决“重复问题成本和延迟”。

但小项目常见问题也很明显：维护不稳定、测试少、license 不清、工程治理弱、权限和审计缺失。学习它们时，目标不是把仓库搬进项目，而是提炼设计模式。

## 2. 三段式训练

### 第一段：读代码，但不背代码

你要让 Agent 带你读四件事：

| 要读什么 | 你要问什么 |
|---|---|
| 输入输出 | 这个项目接收什么，返回什么，失败怎么表达？ |
| 核心假设 | 它认为 RAG 失败主要来自哪一层？ |
| 关键变量 | 哪些参数最影响效果？ |
| 缺失治理 | 它缺少权限、审计、评测、版本还是回滚？ |

推荐提问：

```text
请把这个小项目按“输入 -> 处理 -> 输出 -> 失败路径 -> 适用边界”解释给我。
我不需要记代码，请帮我提炼它解决的 RAG 问题、核心假设和不能直接上生产的原因。
```

### 第二段：复现实验，而不是复刻仓库

你要复现的是“问题是否真的被解决”，不是复刻目录结构。

| 复现对象 | 说明 |
|---|---|
| baseline 失败 | 不用该项目时会怎样失败 |
| 最小变量 | 只打开该项目代表的一个优化 |
| 对照指标 | hit@k、context precision、faithfulness、latency、cost |
| 反例 | 哪些问题上它没有提升，甚至退化 |

推荐提问：

```text
请把这个小项目改造成一个 RAG 实验：baseline 是什么，变量是什么，指标是什么，失败样例是什么，什么情况下不该上线。
```

### 第三段：重写最小设计，而不是记实现

这里的“重写”不是让你写代码，而是让你用自己的话重新设计：

| 设计层 | 你要产出 |
|---|---|
| 契约 | 输入、输出、错误、版本字段 |
| 策略 | 什么时候启用，什么时候禁用 |
| 评测 | 哪些 case 证明它有效 |
| 安全 | 是否影响权限、注入、数据泄漏 |
| 治理 | 如何发布、回滚、观测 |

推荐提问：

```text
请把这个小项目的能力抽象成我们的 Agent Runtime Gateway 设计卡片：契约、启用条件、评测集、风险和上线门禁。
```

## 3. 小项目评分表

| 维度 | 0 分 | 1 分 | 2 分 |
|---|---|---|---|
| 问题清晰度 | 不知道解决什么 | 大概有方向 | 明确对应 RAG 失败层 |
| 实验可复现 | 没有样例 | 有 demo | 有数据集、指标或脚本 |
| 方法边界 | 夸大万能 | 有少量限制 | 明确适用/不适用场景 |
| 工程可迁移 | 强绑定项目 | 可拆一点 | 可抽象成契约和策略 |
| 安全治理 | 未考虑 | 有基本限制 | 可接入 ACL、audit、eval gate |
| 维护风险 | 停更或 license 不清 | 一般 | 活跃且 license 清楚 |

建议：

- 10 分以上：可以进入实验候选。
- 7 到 9 分：只参考模式，不进入主线。
- 6 分及以下：只作为概念学习，不纳入工程路线。

## 5. 最重要的反思

如果一个小项目不能被转化为“契约、策略、评测、治理”，它就不适合直接进入工业级 Agent 框架。

你需要学会的不是“这个库怎么写”，而是：

```text
它解决了什么失败？
它的假设什么时候成立？
我怎样证明它有效？
它进入企业系统前还缺什么护栏？
```
