# 全链路故障与修复手册

用途：按“症状 -> 定位证据 -> 根因 -> 修复 -> 回归”排查 OpsPilot。修复任何事故后，都必须把最小复现加入 `21-测试` 或 `22-评测集`。

| 环节 | 典型问题 | 首先检查 | 常见根因 | 修复 | 必加回归 |
|---|---|---|---|---|---|
| Request Gateway | 未认证/跨租户请求进入 Runtime | auth audit、tenant 来源 | 信任请求体里的 role/tenant | 身份系统注入事实；默认拒绝 | 伪造 role、tenant mismatch |
| Contract | 坏字段流入下游 | validation error、schema version | 宽松 dict、静默忽略字段 | 严格 schema、版本兼容、typed error | 缺字段、错类型、额外字段 |
| Model Gateway | 超时、限流、输出不可解析 | provider span、retry count | 无 timeout；盲重试；schema 漂移 | deadline、退避+jitter、structured output、fallback 门禁 | timeout、429、malformed output |
| Planner | 死循环或计划膨胀 | step count、重复 action | 无预算/终止条件；隐藏状态 | max steps、budget、显式 state、loop detector | 重复 tool call、无进展循环 |
| Tool Registry | 调错工具或版本 | registry version、manifest | 工具描述含糊；运行时动态漂移 | 版本化 registry、能力最小化、契约测试 | unknown/disabled/version mismatch |
| Tool Gateway | 越权或参数注入 | policy decision、sanitized args | 把模型输出当权限；未校验参数 | 系统事实驱动 policy、严格参数 schema | prompt 声称管理员、路径/SQL 注入 |
| Approval | 绕过、重放或审批错对象 | approval subject、operation id | 审批只绑定用户文本；无过期 | 绑定 call hash/版本/范围；TTL；一次性 token | 修改审批后参数、过期审批 |
| Side Effect | 重复退款/发信 | operation ledger、provider id | 超时后直接重试；无幂等键 | idempotency key、查询后重试、outbox/ledger | 成功后断网、重复 resume |
| Runtime State | 非法跳转、任务卡死 | state/event timeline | 状态散落在 prompt；无转换表 | 显式状态机、合法转换、timeout/cancel | waiting 直接 completed、stuck run |
| Checkpoint | 恢复后丢状态或重复执行 | checkpoint version/checksum | 只存聊天；写入非原子 | 保存业务状态与副作用引用；事务写入 | crash before/after tool commit |
| RAG Ingestion | 文档缺失、重复、版本混乱 | corpus manifest、chunk lineage | 无解析质量检查；无 content hash | 版本化文档/chunk/index；去重与校验 | 空文档、重复文档、坏编码 |
| Retrieval | 找不到关键政策 | retrieval eval、query、filters | chunk 不合理；过滤过严；索引陈旧 | error analysis 后再选 rewrite/hybrid/rerank | hard query、同义词、日期条件 |
| Tenant ACL | 检索到其他租户 | retrieved metadata、policy audit | 生成后过滤；缓存键缺 tenant | 检索前强制 ACL；tenant-aware cache | 跨租户 query 和 cache collision |
| Citation | 有答案但引用不支持 | claim-citation mapping | 只返回相似 chunk；引用版本错 | claim-level citation check、拒答阈值 | unsupported claim、stale citation |
| Memory Write | 错误事实或敏感数据入库 | source、write decision、PII flag | 自动写全部对话；模型总结当事实 | write gate、来源/置信度、最小化、人工纠正 | prompt 注入记忆、PII write |
| Memory Read | 过期/跨租户/已删数据仍召回 | tenant/TTL/delete tombstone | 向量库与主库删除不同步 | read filter、同步删除、召回回归 | TTL expiry、delete、tenant isolation |
| Context | 关键信息丢失或成本暴涨 | token allocation、context trace | 无优先级；重复塞全文 | context budget、去重、摘要、ablation | 长对话、重复文档、关键约束保留 |
| Eval Dataset | 指标虚高或不稳定 | dataset version、split、variance | 泄漏；只有简单题；judge 未校准 | 隔离集、hard/adversarial cases、多次 trial | known incident、unseen holdout |
| LLM Judge | 偏置、自洽或漂移 | judge prompt/model/version | 单 judge；无人工基准 | rule-first、人工校准、pairwise/多 judge | 刻意讨好 judge、格式偏置 |
| Trace | 无法定位或泄密 | span coverage、attributes | correlation id 丢失；记录 prompt/secret | 统一 IDs、脱敏、采样/保留策略 | missing span、secret canary |
| Audit | 无法证明谁批准什么 | actor/action/resource/time/outcome | 把 debug log 当 audit；可修改 | append-only 语义、最小字段、访问控制 | policy allow/deny、approval、admin change |
| Cost/Latency | P95 暴涨 | per-step tokens/latency、queue time | 上下文膨胀；串行工具；无限 retry | budget、缓存、并行只读、模型路由、deadline | long context、slow provider、retry storm |
| Prompt Injection | 文档指令控制工具 | untrusted-source marker、tool trace | 指令与数据混合；过度权限 | 数据/指令分离、least privilege、approval、egress policy | direct/indirect injection corpus |
| MCP | 恶意/漂移 server | server identity、tool diff、network | 自动信任新工具；长期凭据 | 准入、pin version、allowlist、隔离、kill switch | tool rug-pull、schema change、exfiltration |
| Sandbox | 文件/网络/进程逃逸 | sandbox audit、egress logs | 仅容器无额外限制；挂载/凭据过宽 | 分级隔离、只读 FS、网络策略、短期凭据 | forbidden path/domain/process |
| Multi-agent | 循环、冲突、错误多数票 | handoff graph、budget、ownership | 角色不清；共享写资源；按票合并 | max depth、资源锁、证据合并、supervisor | circular handoff、same-file conflict |
| Storage | 数据丢失或迁移失败 | transaction/migration logs | 非事务多写；无 backward compatibility | transaction/outbox、expand-contract、backup/restore drill | partial commit、old/new schema |
| Release | 坏 prompt/model/index 上线 | release manifest、eval gate | 只版本化代码；关键失败不阻塞 | 锁全版本、canary、automatic rollback | bad prompt、stale index、failed migration |
| Incident Response | 修好但再次发生 | postmortem actions、regression link | 只修症状；无 owner/deadline | 五问根因、永久防线、回归门禁 | 用事故输入重放全链路 |

## 通用排查顺序

1. 固定 `run_id`、`trace_id`、release version 和输入，禁止先“多跑几次看看”。
2. 判断错误属于输入、模型、检索、工具、状态、策略、存储还是基础设施。
3. 找到最后一个正确事件和第一个错误事件，缩小到单一边界。
4. 写最小失败用例，在修复前证明它确实失败。
5. 进行最小修复，运行相邻模块和全链路回归。
6. 更新 eval、告警或发布门禁，确保相同问题不能静默回归。
7. 记录剩余风险、owner 和截止日期。

## 不应采用的“修复”

- 只改 prompt，未修权限、状态或数据边界。
- 对所有错误无限重试。
- 为了通过平均分删除困难 eval case。
- 关闭审计或安全检查换取延迟。
- 引入多 Agent 掩盖单 Agent 的契约和工具问题。
- 使用更大模型替代根因分析。
