## 2. 前沿结论

| 方向 | 结论 | 教学含义 |
|---|---|---|
| Agentic RAG | RAG 从固定 chain 变成计划、路由、查询改写、多源检索、验证、重试闭环 | 不能一开始就教“向量库 + topK”，要教检索决策 |
| GraphRAG | 适合关系密集、多跳、全局主题和跨文档归纳 | 放在普通 RAG 之后，不进入第一版 MVP |
| LightRAG | 用更轻的 graph + vector 组合降低 GraphRAG 成本 | 适合作为 GraphRAG 的轻量实验 |
| Self-RAG / CRAG | 适合检索质量不稳定，需要判断是否检索、证据是否支持、是否纠正 | 放入失败恢复和 Agentic RAG 章节 |
| HyDE / query rewrite | 适合短查询、零样本、语义缺口和跨语言 | 作为检索前优化，而不是生成答案技巧 |
| Hybrid search + rerank | 关键词 + 语义 + reranker 是高质量检索的常见生产候选基线，需要按成本、延迟和任务风险路由 | Phase 6 不能只做 vector search，也不能无条件启用昂贵 reranker |
| Late interaction / ColBERT | 适合高精度、法律、金融、代码、PDF 页级检索 | 作为高精度检索实验，不默认上生产 |
| Long-context RAG | 长上下文不是 RAG 替代品，适合小量超长材料和需要完整上下文的问题 | 需要 route：先判断 RAG、LC 或两者结合 |
| Memory + RAG | 长线 Agent 需要长期记忆、任务记忆和知识库检索区分 | 记忆写入门禁必须和 RAG retrieval policy 配合 |
| RAG eval/observability | 必须覆盖 retrieval、faithfulness、answer、trace、online drift | 评测不是最后一步，而是每个优化的证据 |
| RAG security | Prompt injection、poisoning、tenant escape、vector weakness 必须单独测 | RAG 不是安全读知识库，需要红队和 ACL |
