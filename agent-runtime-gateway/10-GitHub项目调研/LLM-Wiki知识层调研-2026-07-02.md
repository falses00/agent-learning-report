# LLM-Wiki 知识层调研

日期：2026-07-02  
任务：为工业级 Agent/RAG 学习项目补充可扩展、易维护的知识层  
调用角色：researcher subagent，只读调研  
结论状态：建议采用“知识治理模式”，不复制外部仓库内容

## 1. 本轮结论

建议把 `llm-wiki` 相关思想加入项目，但它在本项目里不是“再做一个资料库”，而是一套知识治理层：

```text
原始来源 raw_sources
-> 可审核知识卡 KnowledgeCard
-> 版本记录 KnowledgeVersion
-> 来源审核 SourceReview
-> 导入队列 ImportQueue
-> 课程页面 / RAG 诊断 / 复述低分回流
```

采用原则：

- 外部项目只作为分类法、学习路径、治理模式、RAG 诊断标签和前端模块参考。
- 不整仓导入。
- 不直接翻译搬运 README、教程正文、图片、notebook 或课程文案。
- 每张知识卡必须有来源、版本、置信度、审核状态和适用 Phase。
- LLM 生成的 wiki/card 默认是草稿，不能直接进入课程主线。

## 4. 采用分级

| 采用方式 | 含义 | 允许做 | 禁止做 |
|---|---|---|---|
| `pattern_only` | 只学习治理模式 | 重写成项目自己的规则 | 复制实现、复制 prompt、复制 README |
| `taxonomy_only` | 只学习分类结构 | 抽象成主题树和术语表 | 搬目录名作为课程正文 |
| `course_reference` | 参考课程组织 | 设计本项目教学顺序 | 复制课程内容、图片、notebook |
| `rag_diagnostic_reference` | 参考 RAG 问题标签 | 转成诊断器问题、指标、eval case | 把外部 notebook 当作本项目答案 |
| `do_not_copy` | 仅保留链接 | 记录来源和风险 | 进入课程正文 |

## 5. 对本项目的真实价值

`llm-wiki` 最有价值的不是“自动生成很多知识页”，而是给本项目补上长期维护机制：

- 新资料进来时，先进入 `ImportQueue`，不直接写进课程。
- 外部来源先做 `SourceReview`，检查 license、更新时间、质量和版权风险。
- 课程知识沉淀为 `KnowledgeCard`，每张卡都可追溯、可过期、可废弃。
- 每次复述低分、RAG 诊断失败或测评回归，都能回流到对应知识卡。
- 前端可以显示知识卡来源、状态、版本和适用 Phase，让用户知道“为什么这么教”。

这能解决当前课程最容易失控的三个问题：

1. 资料越来越多，但不知道哪些可信。
2. GitHub 热点变化快，课程容易过期。
3. 学习者低分反馈如果只在对话里出现，就无法沉淀成下一轮教学改进。

## 7. 进入前端的设计要求

前端要增加一个“知识层 / LLM-Wiki”学习模块，但它仍然是教学模块，不是真实资料抓取系统。

模块必须显示：

- 来源队列 `ImportQueue`。
- 来源审核 `SourceReview`。
- 主题树 `KnowledgeTopic`。
- 知识卡 `KnowledgeCard`。
- 版本记录 `KnowledgeVersion`。
- 新旧版本差异。
- freshness / license / review_status。
- 与 Phase、RAG 诊断、复述低分的映射。

禁止显示：

- 一键导入外部仓库。
- 自动改写课程主线。
- 复制外部 README。
- 把低置信内容标成已通过。
- 不带来源的知识卡。

## 8. 进入课程的最小规则

一条外部资料想进入课程，必须回答 8 个问题：

| 问题 | 通过样子 |
|---|---|
| 它解决哪个学习卡点？ | 能绑定 Phase、Concept 或 RAG 诊断问题 |
| 它来自哪里？ | 有 `KnowledgeSource` 和 URL |
| license 是否可接受？ | 有 `SourceReview.license_result` |
| 是否最新？ | 有 `github_updated_at`、`fetched_at`、`reviewed_at` |
| 是否容易幻觉？ | 有置信度和证据范围 |
| 是否适合中文学习者？ | 有 canonical term 和中文解释 |
| 是否能转成复述题？ | 能生成 `RestatementCard` |
| 是否会影响评测？ | 能映射到 `EvalCase` 或说明不影响 |

任何一项缺失，只能进入证据库，不能进入课程主线。

## 9. 对未来维护的建议

本项目后续维护不要追求“资料全”，而要追求“资料可治理”。

建议每两周做一次小维护循环：

```text
查看低分反馈
-> 查看 RAG / Agent 新资料
-> 进入 ImportQueue
-> 做 SourceReview
-> 更新 KnowledgeCard
-> 更新 RestatementCard / EvalCase
-> 前端显示版本变化
```

真正工业级的学习项目，必须能回答：

```text
这条知识从哪来？
什么时候审过？
为什么可信？
过期后谁负责？
学不会时回流到哪里？
上线前用哪个 eval case 证明？
```

如果回答不了，就不是可维护知识层，只是资料堆积。
