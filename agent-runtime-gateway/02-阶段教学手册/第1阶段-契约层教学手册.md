# 第1阶段-契约层教学手册

生成日期：2026-06-28  
阶段：Phase 1  
目标：先把 Agent 系统里的关键对象结构化，给后续网关、运行时、评估、安全和治理打地基。

## 1. 本阶段为什么重要

企业级 Agent 不能只靠自然语言约束。

先回到工单故事：

```text
客户说：我上个月买的企业套餐无法使用，请帮我退款。
```

如果这是聊天 demo，模型可能直接回答“建议退款”。  
但工业级 Agent 要先把事情拆成结构化事实：

- 这是哪个 Agent 在处理？
- 当前 Agent 可以用哪些工具？
- 它申请调用的工具是什么？
- 这次调用是读政策、查订单，还是执行退款？
- 发起人是谁，属于哪个租户？
- 如果请求被拒绝，错误应该怎么记录？

这些问题如果只靠自然语言，后续 Gateway、Policy、Audit、Eval 都没法稳定接住。

如果没有契约层，后面会出现这些问题：

- Agent 调用工具时参数随意，无法稳定验证。
- 权限系统不知道应该判断哪些事实。
- 错误只有一段文本，无法自动重试、恢复或审计。
- 评估系统无法复现一次失败。
- 多 Agent 协作时，每个角色对同一件事的理解不同。

契约层要做的事，就是把“模糊行为”变成“结构化对象”。

白话说：

```text
Phase 1 是给 Agent 行为办身份证、写申请单、准备权限审查表、定义错误回执。
```

## 2. 最小产物

Phase 1 至少要产出四类 schema：

| 产物 | 作用 |
|---|---|
| `AgentManifest` | 描述 Agent 的身份、版本、模型、工具、策略和评估集 |
| `ToolCall` | 描述一次工具调用申请 |
| `PolicyDecisionInput` | 描述策略判断需要的事实 |
| `ErrorModel` | 描述错误类型、是否可恢复、审计等级和用户可见信息 |

在工单故事里，它们对应：

| 产物 | 工单故事里的意思 |
|---|---|
| `AgentManifest` | 这个工单 Agent 的身份证：它是谁、能用什么模型和工具、默认风险多高 |
| `ToolCall` | Agent 提交的一张工具申请单：我要查退款政策或读取订单 |
| `PolicyDecisionInput` | 权限审查表：谁要对哪个资源做什么操作 |
| `ErrorModel` | 拒绝或失败时的标准回执：为什么失败、能不能重试、要不要审计 |

## 3. AgentManifest 应包含什么

建议字段：

| 字段 | 说明 |
|---|---|
| `agent_id` | Agent 唯一 ID |
| `name` | 人类可读名称 |
| `version` | 版本号 |
| `owner` | 负责人或团队 |
| `model_policy` | 可用模型、fallback、成本限制 |
| `tools` | 可用工具列表 |
| `memory_policy` | 记忆写入和召回规则 |
| `eval_sets` | 发布前必须通过的评估集 |
| `risk_level` | 默认风险等级 |

教学重点：manifest 是 Agent 的“身份证 + 发布配置”，不是 prompt 文本仓库。

## 4. ToolCall 应包含什么

建议字段：

| 字段 | 说明 |
|---|---|
| `tool_name` | 工具名 |
| `tool_version` | 工具版本 |
| `operation_id` | 幂等键，用于防止重复副作用 |
| `arguments` | 结构化参数 |
| `reason` | Agent 解释为什么要调用 |
| `risk_level` | 本次调用的风险等级 |
| `requested_by` | 请求主体 |
| `trace_id` | 链路追踪 ID |

教学重点：ToolCall 是“申请”，不是“执行”。是否执行必须交给 Tool Gateway 和 Policy。

## 5. PolicyDecisionInput 应包含什么

建议字段：

| 字段 | 说明 |
|---|---|
| `principal` | 谁发起请求 |
| `tenant_id` | 属于哪个租户 |
| `workspace_id` | 属于哪个工作区 |
| `tool_name` | 要调用哪个工具 |
| `operation` | 读、写、删除、执行等操作 |
| `resource` | 目标资源 |
| `risk_level` | 风险等级 |
| `context` | 工单类型、时间、来源等上下文 |

教学重点：策略判断要基于事实，不基于 prompt 里的承诺。

## 6. ErrorModel 应包含什么

建议字段：

| 字段 | 说明 |
|---|---|
| `code` | 稳定错误码 |
| `category` | validation、policy、tool、model、runtime、sandbox |
| `recoverable` | 是否可恢复 |
| `retryable` | 是否可重试 |
| `user_message` | 可给用户看的信息 |
| `internal_message` | 内部排查信息 |
| `audit_level` | 审计等级 |

教学重点：错误模型不是为了好看，而是为了自动恢复、评估统计和事故复盘。

## 7. 必须设计的坏输入

Phase 1 不能只测 happy path，至少要测：

- 缺少必填字段。
- 多出未知字段。
- 类型错误，例如 `risk_level` 传数字或脚本文本。
- 工具名不存在。
- `operation_id` 为空。
- 用户把 prompt injection 放进工具参数。
- 低权限主体申请高风险写操作。
- 跨租户资源 ID 被传入。

## 8. 最小样例

你不需要写代码，但要能看懂合法输入和坏输入的区别。

### 8.1 合法 ToolCall

```json
{
  "tool_name": "knowledge.retrieve",
  "tool_version": "2026-07-01",
  "operation_id": "op_read_policy_001",
  "arguments": {
    "query": "enterprise plan refund policy",
    "tenant_id": "tenant_demo",
    "document_scope": "refund_policy"
  },
  "reason": "Need refund policy before advising the ticket handler.",
  "risk_level": "L1",
  "requested_by": {
    "principal": "user_support_123",
    "tenant_id": "tenant_demo"
  },
  "trace_id": "trc_ticket_refund_001"
}
```

为什么合法：

- 有工具名和版本。
- 有 `operation_id`，后续可追踪和防重复。
- 是只读检索，不是直接退款。
- 带了请求主体和租户。
- 有 `trace_id`，后续能审计和排查。

### 8.2 坏输入：缺少 operation_id

```json
{
  "tool_name": "billing.refund",
  "tool_version": "2026-07-01",
  "arguments": {
    "customer_id": "cus_001",
    "amount": 999
  },
  "reason": "Customer asked for refund.",
  "risk_level": "L3",
  "requested_by": {
    "principal": "user_support_123",
    "tenant_id": "tenant_demo"
  },
  "trace_id": "trc_ticket_refund_002"
}
```

预期拒绝：

```json
{
  "code": "tool_call.missing_operation_id",
  "category": "validation",
  "recoverable": true,
  "retryable": false,
  "user_message": "The refund request cannot be submitted yet.",
  "internal_message": "High-risk ToolCall is missing operation_id.",
  "audit_level": "high"
}
```

为什么必须拒绝：

```text
退款是高风险写操作。
没有 operation_id，重试时可能重复退款。
```

### 8.3 坏输入：跨租户资源

```json
{
  "principal": "user_support_123",
  "tenant_id": "tenant_demo",
  "workspace_id": "ws_demo",
  "tool_name": "knowledge.retrieve",
  "operation": "read",
  "resource": {
    "tenant_id": "tenant_other",
    "document_id": "refund_policy_private"
  },
  "risk_level": "L1",
  "context": {
    "ticket_id": "ticket_7788"
  }
}
```

预期拒绝：

```json
{
  "code": "policy.cross_tenant_resource_denied",
  "category": "policy",
  "recoverable": false,
  "retryable": false,
  "user_message": "You do not have access to this resource.",
  "internal_message": "principal tenant does not match resource tenant.",
  "audit_level": "critical"
}
```

为什么必须拒绝：

```text
即使只是读取知识库，也不能跨租户召回。
权限问题不能交给 prompt 解释，必须由 Policy 拒绝并写 audit。
```

## 9. 分层复述模板

### 一句话版

```text
Phase 1 把 Agent 的模糊行为变成结构化契约。
```

### 业务例子版

```text
在退款工单里，Agent 不能只说“我要退款”。
它必须提交 ToolCall，Policy 要根据 principal、tenant、resource、operation 判断能不能做。
如果缺少 operation_id 或跨租户访问，就要返回结构化 ErrorModel。
```

### 卡住自查版

```text
我能解释 AgentManifest 是什么吗？
我能解释 ToolCall 为什么只是申请，不是执行吗？
我能说出一个坏输入吗？
我能说出坏输入对应的 ErrorModel 吗？
我能说明为什么 prompt 不能替代 Policy 吗？
```

## 10. 完成标准

Phase 1 完成时，必须能证明：

- schema 能接受合法输入。
- schema 能拒绝坏输入。
- 错误输出是结构化的。
- 每个字段都能解释为什么存在。
- Phase 2 的最小运行时和 Phase 3 的 Request Gateway / Tool Gateway 可以直接使用这些契约。

## 11. 本阶段复述

完成后你应该能说清楚：

```text
Phase 1 解决了 Agent 行为不可结构化的问题。
我们定义了 AgentManifest、ToolCall、PolicyDecisionInput 和 ErrorModel。
它让后续网关、权限、评估、审计和恢复都有统一输入。
我用合法输入和坏输入验证 schema 是否可靠。
```

如果你只能背出四个 schema 名字，但说不出退款工单里它们各自防什么事故，就不要进入 Phase 2。
