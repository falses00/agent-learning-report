最近更新：2026-07-01  
用途：每个阶段完成前，用清单判断是否真的可以进入下一步。

## 使用方法

本项目有两套验收，不要混用：

| 模式 | 当前是否默认 | 看什么证据 | 不要求什么 |
|---|---|---|---|
| Design-only | 是 | 设计卡、失败场景、评审问题、验收门禁、用户复述 | 不要求写代码、跑命令、创建源码目录 |
| Implementation-later | 后续可选 | 代码、测试、trace、eval report、audit、CI gate | 不允许只停留在概念文档 |

Design-only 不降低工业级标准，只改变证据形态：当前先证明你会设计、会评审、会判断风险；后续工程阶段再把这些设计落成代码和自动化验证。

当前阶段只用 Design-only 作为通过依据；Implementation-later 是后续工程 backlog 和发布门禁，不要求当前用户写代码、建目录或跑测试。

Design-only 阶段结束时必须提交 4 类证据，任一缺失不得进入下一阶段：

- [ ] 设计卡：说明这一阶段防止哪类 Agent 失控，哪一层负责拦截、记录、恢复或评测。
- [ ] 失败样例：说明缺少这一层会造成什么生产事故。
- [ ] 评审问题：至少 3 个能暴露坏设计的问题。
- [ ] 复述记录：评分 >=3，且答案同时包含事故、负责层和验收证据。

后续 Implementation-later 工程证据可以先列为 backlog，但不能替代当前 Design-only 证据。

## 可视化辅助验收规则

[学习可视化前端 PRD](../01-产品需求与路线/学习可视化前端PRD.md) 可以作为课程辅助，但不能单独让任何阶段过关。

可视化证据只有同时满足以下条件，才可计入 Design-only 验收：

- [ ] 页面明确标识 `学习模式 Learning mode / 模拟数据 Mock data / 不执行真实操作 No real execution / 来源文件 Source files`。
- [ ] 页面绑定具体 Phase，而不是泛泛展示 Agent 概念。
- [ ] 页面有一个坏设计或事故样例。
- [ ] 页面能指向对应课程源文件或蓝图文件。
- [ ] 学习者完成复述评分，且评分 >=3。
- [ ] 复述答案同时包含事故、负责层和验收证据。
- [ ] 主要学习文案是中文，关键工程术语有英文括注。
- [ ] 低于 3 分的复述能生成卡点标签和回流动作。

以下情况直接判失败：

- [ ] 学习者以为页面已经能连接真实生产系统。
- [ ] 页面出现真实凭据、真实企业数据、真实工具执行或真实发布入口。
- [ ] 页面只有漂亮图形，没有坏设计、停止条件或复述题。
- [ ] 页面替代了阶段手册、Design-only 产物包或阶段验收清单。
- [ ] 页面主要内容全英文，导致半懂学习者无法直接复述。
- [ ] 低分只被记录，没有重讲任务、内容修订或下次复习安排。

统一复述评分下限：

| 分数 | 判断 |
|---|---|
| 0-2 | 不进入下一阶段，先重讲或重画 |
| 3 | 可进入下一阶段，但要记录卡点 |
| 4-5 | 可以进入下一阶段，并把好表达沉淀到课程文档 |

## Phase 0：课程总览与路线审核

Design-only 验收：

- [ ] 用户能说明为什么先做路线、PRD、调研和门禁，而不是直接写 demo。
- [ ] 用户能解释 Runtime、Gateway、Tool、Memory、Eval、Observability、Sandbox、Governance 的总边界。
- [ ] GitHub 调研有日期、来源、风险和采用方式。
- [ ] 需求追踪矩阵能把需求映射到阶段和验收标准。
- [ ] README 能告诉读者先读什么、当前如何提问、下一步是什么。

Implementation-later 工程门禁：

- [ ] 若进入工程阶段，所有需求 ID 都能映射到文件、测试或评测集。
- [ ] 文档链接、JSON 快照和目录结构可被自动校验。

## Phase 1：契约层

Design-only 验收：

- [ ] 用户能说明 `AgentManifest`、`ToolCall`、`PolicyDecisionInput`、`ErrorModel` 分别防止什么失控。
- [ ] 用户能解释 `RetrievalTool`、`RAGRequest`、`RAGResult`、`Citation`、`ContextPack`、`RAGEvalCase` 的责任。
- [ ] 用户能设计合法输入、缺字段、错类型、未知工具、注入文本、跨租户资源等坏输入样例。
- [ ] 用户能说明契约层为什么不是背字段，而是治理边界。
- [ ] RAG 坏输入覆盖缺少 tenant、缺少 trace、top_k 过大、数据等级越权、跨租户结果和无版本结果。

Implementation-later 工程门禁：

- [ ] 定义可运行 schema，并包含版本字段、迁移策略占位、bad-case fixtures 和 ErrorModel。
- [ ] 合法输入能通过 schema。
- [ ] 坏输入能被拒绝或结构化报错。
- [ ] 字段命名、错误码和风险等级稳定，可被 Phase 2 复用。

## Phase 2：最小运行时

Design-only 验收：

- [ ] 用户能画出 `Run -> Step -> Event -> Checkpoint`。
- [ ] 用户能说明为什么聊天记录不能替代运行状态。
- [ ] 用户能推演一次工具失败、模型循环或超时失控时，Runtime 如何停止、记录和恢复。
- [ ] 用户能解释 `trace_id` 为什么必须跨 run、step、tool、policy 传递。

Implementation-later 工程门禁：

- [ ] 有 `Run`、`Step`、`State` 的最小结构。
- [ ] mock model 能产生结构化响应或工具调用意图。
- [ ] mock tool 能返回成功和失败结果。
- [ ] 工具失败能转成结构化 `ErrorModel`。
- [ ] 至少有一个正常 run 和一个失败 run 的测试。

## Phase 3：Agent 网关与工具治理

Design-only 验收：

- [ ] 用户能画出 `Request Gateway -> Runtime -> Tool Gateway -> Policy -> Audit`。
- [ ] 用户能指出 prompt 不能决定权限，权限必须在结构化网关和策略层判断。
- [ ] 用户能设计一个“只读允许、写操作拒绝、越权审计”的工具治理场景。
- [ ] 用户能解释 Tool Gateway 不可被绕过的原因。

Implementation-later 工程门禁：

- [ ] Request Gateway 能生成 `request_id`、`thread_id`、`trace_id`。
- [ ] Model Gateway 能执行 provider routing、fallback 或预算限制的最小策略。
- [ ] Tool Gateway 能拦截并校验工具调用。
- [ ] Policy 能允许只读操作、拒绝越权写操作。
- [ ] Audit Event 能记录主体、工具、参数摘要、策略结果和工具结果。

## Phase 4：长线任务与断点恢复

Design-only 验收：

- [ ] 用户能解释 checkpoint、resume、retry、cancel、fork 和 HITL 分别解决什么长线任务风险。
- [ ] 用户能推演重复 `operation_id` 为什么不能产生重复副作用。
- [ ] 用户能说明任务状态、人工审批和工具副作用之间的关系。
- [ ] 用户能设计一次中断恢复的时间线。

Implementation-later 工程门禁：

- [ ] checkpoint 能保存并恢复。
- [ ] retry 不破坏幂等。
- [ ] cancel/fork/resume 的行为有明确规则。
- [ ] human-in-the-loop 能暂停和继续任务。
- [ ] checkpoint resume、重复 operation_id、approval resume、max step guard 有测试。

## Phase 5：MCP 与工具注册

Design-only 验收：

- [ ] 用户能区分 MCP 解决的是工具协议，不解决企业权限、审计、租户和凭据治理。
- [ ] 用户能为一个 MCP server 设计来源、版本、能力、风险、凭据、禁用路径和回归样例。
- [ ] 用户能解释为什么公共 registry 不能直接等同企业可信 registry。
- [ ] 用户能说明工具 schema、工具风险等级和凭据代理的边界。

Implementation-later 工程门禁：

- [ ] Tool Registry 能记录工具 schema、版本、owner、风险等级。
- [ ] MCP adapter 能调用至少一个 mock 或本地 MCP-like 工具。
- [ ] 凭据不暴露给 Agent prompt。
- [ ] 工具结果进入过滤和审计。
- [ ] MCP server 有准入、health check、allowlist、disable path 和降级策略。

## Phase 6：记忆系统

Design-only 验收：

- [ ] 用户能说明知识库、会话记忆、长期记忆和事件记忆的边界。
- [ ] 用户能设计记忆写入门禁：来源、权限、TTL、删除、纠错和审计。
- [ ] 用户能设计一个记忆污染、过期召回或跨租户召回的失败样例。
- [ ] 用户能解释为什么默认写入长期记忆是危险设计。

Implementation-later 工程门禁：

- [ ] 区分短期、长期、事件、语义记忆。
- [ ] 记忆写入需要来源、权限、过期和删除规则。
- [ ] 跨租户记忆不能被召回。
- [ ] 错误事实能撤销或标记失效。
- [ ] 记忆评测覆盖命中率、污染率、过期信息和跨租户泄漏。

## Phase 7：测评审核与红队

Design-only 验收：

- [ ] 用户能区分普通低分、发布阻塞、安全阻塞和需要人工审批的失败。
- [ ] 用户能设计 golden、red team、regression、trajectory、memory、RAG 评测样例。
- [ ] RAG 评测集设计覆盖 smoke、hard retrieval、citation、freshness、security、tenant ACL、memory boundary。
- [ ] RAG JSONL 示例能说明每条 case 的失败类型、指标和是否阻塞发布。
- [ ] 用户能说明为什么评测不能只看最终答案。

Implementation-later 工程门禁：

- [ ] 有 golden dataset。
- [ ] 有 red team dataset。
- [ ] 评估覆盖准确率、工具选择、越权、注入和记忆污染。
- [ ] 评估覆盖 Agent 轨迹、重复工具调用、恢复能力和成本。
- [ ] 模型或 prompt 变更必须跑回归，critical red team 失败必须阻塞发布。

## Phase 8：可观测性与审计

Design-only 验收：

- [ ] 用户能设计一次请求的 span 列表，覆盖模型、工具、策略、记忆、RAG 和错误。
- [ ] 用户能区分调试日志、trace、metrics 和 audit event 的用途。
- [ ] 用户能说明一次失败如何从 trace 定位到具体模块。
- [ ] 用户能设计成本、延迟、失败率、恢复率和评估通过率的观测字段。

Implementation-later 工程门禁：

- [ ] 每次请求有完整 trace。
- [ ] 模型调用、工具调用、策略判断、记忆读写都有 span。
- [ ] 关键指标包含成本、延迟、失败率、恢复率、评估通过率。
- [ ] 审计日志和调试日志边界清楚。
- [ ] 支持至少一次 trace replay 或运行复盘。

## Phase 9：安全隔离与沙箱

Design-only 验收：

- [ ] 用户能按工具风险选择 no sandbox、restricted sandbox、gVisor/microVM 或人工审批。
- [ ] 用户能推演文件、网络、环境变量、secret 和进程逃逸的隔离边界。
- [ ] 用户能说明为什么 Docker 不应被当成唯一安全边界。
- [ ] 用户能设计一个越权、逃逸或数据外传方向的失败样例。

Implementation-later 工程门禁：

- [ ] 工具有风险分级。
- [ ] 高风险工具进入 sandbox 或人工审批。
- [ ] 文件系统、网络、进程和环境变量有隔离策略。
- [ ] 沙箱失败能结构化返回。
- [ ] 高风险工具没有 sandbox/approval 不得启用。

## Phase 10：多 Agent 协同

Design-only 验收：

- [ ] 用户能为 planner、executor、reviewer、verifier 设计输入、输出、停止条件和证据要求。
- [ ] 用户能解释多 Agent 分歧为什么按证据合并，而不是按投票合并。
- [ ] 用户能说明 handoff schema、shared state、resource lock、arbiter 和 loop guard 的职责。
- [ ] 用户能设计一个重复任务、资源冲突或协同死循环的失败样例。

Implementation-later 工程门禁：

- [ ] supervisor、worker、reviewer、verifier 的职责清楚。
- [ ] 每个 subagent 有输入、输出、停止条件。
- [ ] handoff schema 包含任务、状态、证据、剩余风险和停止条件。
- [ ] 不允许多个 agent 同时编辑同一文件范围。
- [ ] resource lock、arbiter、loop guard 有 trace 和失败路径测试。

## Phase 11：治理控制台与发布

Design-only 验收：

- [ ] 用户能画出“变更 -> 评测 -> 门禁 -> 发布 -> 回滚”的治理链路。
- [ ] 用户能说明 Agent、工具、模型、策略、评估集和版本如何一起发布。
- [ ] 用户能解释 release audit packet、rollback plan、production readiness checklist 的区别。
- [ ] 用户能说明 RAG 发布为什么要绑定 document、parser、chunker、index、embedding、reranker、context pack、prompt、eval set 版本。

Implementation-later 工程门禁：

- [ ] 能管理 Agent、工具、模型、策略、评估集和版本。
- [ ] 发布前能看到评估结果和风险。
- [ ] 支持回滚、禁用、审计查询。
- [ ] 支持租户和角色权限。
- [ ] 关键治理操作有审计事件。
- [ ] release audit packet、rollback plan、production readiness checklist 齐全。
