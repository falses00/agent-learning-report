# 第1课逐题主持脚本

日期：2026-07-02  
用途：把 v0.4 原型、学习效果验证计划和评分表变成真实课堂对话流程。  
边界：Design-only，不写代码，不记代码，不伪造分数。

## 1. 主持人规则

主持人每次只问一个主题，不连续灌输。

固定节奏：

```text
看图前提问
-> 等学习者回答
-> 0-5 分初评
-> 引导打开原型对应页面
-> 用退款工单故事重讲
-> 看图后复述
-> 坏设计判断题
-> 记录分数、缺失项、回流对象
-> 决定继续或停下重讲
```

禁止：

- 不要替学习者填写分数。
- 不要因为页面能点就判定学会。
- 不要把原型称为生产控制台。
- 不要要求学习者写代码。
- 不要在低于 3 分时进入下一题。

## 2. 评分速查

| 分数 | 判断 |
|---:|---|
| 0 | 完全说不出，或把 mock 当生产 |
| 1 | 只背术语 |
| 2 | 能说大意，但缺事故、负责层或证据 |
| 3 | 能说出事故、负责层、最小证据 |
| 4 | 能指出坏设计和停止条件 |
| 5 | 能主动提出评测、审计、回归和实现门禁 |

## 4. 开场白

```text
我们现在开始第 1 课。
本课不写代码，也不背代码。
你只需要用自己的话判断：事故是什么，哪一层负责，最小证据是什么。

如果你回答低于 3 分，我会停下来重讲，不会推进下一题。
```

## 5. 逐题脚本

### 1. 建议与执行分离

打开位置：C 首页 / A 全链路。

看图前提问：

```text
为什么模型建议不能直接退款？
```

看图后必须说出：

```text
事故：越权退款。
负责层：Request Gateway / Policy / Tool Gateway。
最小证据：学习边界、ToolCall 申请、PolicyDecision、AuditEvent。
```



## 7. 总结判定

课堂结束后只按这个规则判定：

```text
13 个 P0 主题中，看图后 >=3 的数量：__/13
RAG 三类是否全部 >=3：是 / 否
是否出现生产误读：是 / 否
是否所有低分都有回流对象：是 / 否
最终判定：Go / Conditional Go / No-Go
```

如果最终判定不是 Go，不进入 Phase 1。
