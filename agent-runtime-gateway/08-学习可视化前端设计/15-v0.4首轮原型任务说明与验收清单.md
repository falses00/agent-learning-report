# v0.4 首轮原型任务说明与验收清单

日期：2026-07-02  
状态：Product Design 低保真可点击原型任务草案  
用途：把待用户显式确认的 brief 转成 Product Design 可执行任务、页面范围、交互清单和验收标准

> 门禁声明：本文件不是开始原型生成的授权。必须先由用户显式确认 [v0.4 低保真可点击原型 Brief 回放](14-v0.4低保真可点击原型Brief回放.md)，才能进入 Product Design ideate / prototype。

## 1. 本轮目标

做一个低保真可点击教学原型，让学习者能通过 6 个页面理解：

```text
Agent 不是聊天框
Agent 是受控长线运行系统
RAG 不是搜到就可信
记忆、工具、评估、发布都要有门禁和证据
```

本轮原型不追求功能完整，追求教学判断完整。

## 2. 首轮 6 页结构

```text
C 首页 / 学习路线地图
-> A 全链路分层控制台
-> B 事故推演工作台
-> RAG 问题诊断器
-> LLM-Wiki 知识层
-> 证据与门禁页
```

每页都必须有统一顶部边界条：

```text
学习模式 Learning mode
模拟数据 Mock data
不执行真实操作 No real execution
来源文件 Source files
```

## 3. 页面任务清单

### 3.1 C 首页 / 学习路线地图

目标：降低入口认知负担，让用户先知道自己在哪。

必须显示：

- 当前 Phase。
- 当前学习任务。
- 当前事故样例。
- 复述评分。
- 卡点标签。
- 下一步。
- source_files。

主行动：

- 查看受控链路。
- 查看事故样例。
- 开始复述练习。

坏设计状态：

```text
用户复述低分，却仍然允许进入下一阶段。
```

验收问题：

```text
为什么 Agent 的建议不等于系统执行许可？
```

### 3.2 A 全链路分层控制台

目标：让用户看到工业级 Agent 的受控链路。

必须显示节点：

- 用户请求 User Request。
- 请求网关 Request Gateway。
- 运行时 Runtime。
- 模型网关 Model Gateway。
- 工具网关 Tool Gateway。
- 策略 Policy。
- RAG。
- Trace / Audit。
- Eval Gate。

主行动：

- 选中节点。
- 切换坏设计路径。
- 查看证据面板。
- 提交复述。

坏设计状态：

```text
模型建议直接调用退款工具，绕过 Tool Gateway / Policy / Audit。
```

验收问题：

```text
为什么 ToolCall 是申请，不是执行？
```

### 3.3 B 事故推演工作台

目标：通过事故倒推负责层、拦截点和证据。

默认事故：

```text
重复退款
```

必须显示：

- 事故摘要。
- 可能原因。
- 事故链路。
- 应拦截层。
- 阻塞判断。
- 低分回流动作。

主行动：

- 选择原因。
- 查看事故链路。
- 判断是否阻塞。
- 回看卡点。

坏设计状态：

```text
retry 没有 operation_id，导致重复副作用。
```

验收问题：

```text
为什么 retry 不能重复副作用？
```

### 3.4 RAG 问题诊断器

目标：把 RAG 从“搜索技巧”提升为“有证据和门禁的系统能力”。

必须显示三类问题：

| 问题 | 必须讲清 |
|---|---|
| 找不到 | 不是简单加 top_k，要看 hard retrieval recall、query rewrite、chunk boundary |
| 找错租户 | 必须 release blocking，因为涉及 tenant ACL 和数据隔离 |
| 引用不可信 | 检索命中不等于可信答案，要看 citation、freshness、SourceReview |

主行动：

- 切换三类问题。
- 查看指标。
- 查看 eval case。
- 判断是否阻塞发布。

坏设计状态：

```text
把检索命中当成可信事实。
```

验收问题：

```text
为什么检索命中不等于可信答案？
```

### 3.5 LLM-Wiki 知识层

目标：让用户知道 GitHub / 外部资料不能直接变成课程事实。

必须显示链路：

```text
ImportQueue
-> SourceReview
-> KnowledgeCard
-> KnowledgeVersion
```

必须显示：

- license_status。
- review_status。
- adoption_mode。
- source_files。
- confidence_score。
- version_id。

主行动：

- 查看来源。
- 审查来源。
- 查看知识卡。
- 判断是否进入课程。

坏设计状态：

```text
因为 GitHub star 高，就直接复制进课程正文。
```

验收问题：

```text
为什么 GitHub 热门资料不能直接进入课程？
```

### 3.6 证据与门禁页

目标：合并首轮 Trace / Audit / Eval / Release 学习模拟，避免页面膨胀。

必须显示：

- mock run timeline。
- trace span。
- audit event。
- eval case matrix。
- release packet。
- rollback plan。

主行动：

- 选中 trace。
- 查看 audit 证据。
- 查看 eval gate。
- 判断 release packet 是否通过。

坏设计状态：

```text
eval critical 失败，但仍然显示可以发布。
```

验收问题：

```text
为什么 eval 不是分数装饰，而是 release blocking 门禁？
```

## 5. 交互状态清单

每页至少包含：

- default。
- selected。
- bad_design。
- blocked。
- low_score。
- empty。
- error。
- mobile_collapsed。

低保真原型可以只做轻交互，但必须能体现：

- 点击节点后右侧证据改变。
- 切换坏设计后出现事故链路。
- 低分后出现回流对象。
- RAG 三类问题可切换。
- LLM-Wiki 来源审核可切换状态。
- Eval critical 失败时 release blocked。

## 6. 文案规范

按钮必须是中文学习动作。

推荐按钮：

- 查看受控链路。
- 查看事故链路。
- 判断是否阻塞。
- 查看证据。
- 提交复述。
- 回看卡点。
- 查看来源文件。
- 查看评测样例。

禁止按钮：

- 执行退款。
- 发布上线。
- 审批通过。
- 连接生产。
- 写入真实记忆。
- 调用真实工具。

## 7. 验收标准

Product Design 原型输出后，用以下标准验收：

| 验收项 | 通过标准 |
|---|---|
| 30 秒定位 | 用户能指出当前 Phase、当前任务和下一步 |
| 事故理解 | 用户能说出页面在防什么事故 |
| 负责层 | 用户能指出 Gateway / Runtime / Policy / RAG / Memory / Eval / Trace / Audit 等负责层 |
| 证据 | 用户能指出 source_files、trace、eval case、audit event 或 mock 数据 |
| RAG | 用户能区分找不到、找错租户、引用不可信 |
| LLM-Wiki | 用户知道外部资料需要 SourceReview 和 KnowledgeVersion |
| 低分回流 | 用户知道低分后回到哪个 Concept / FailureCase / EvalCase / KnowledgeCard |
| 安全边界 | 用户不会把原型误认为生产系统 |

## 8. No-Go 条件

出现任一情况，不进入下一阶段：

- 页面像真实生产后台。
- 页面出现真实执行、真实发布、真实审批、真实退款。
- RAG 页暗示“检索命中 = 可信事实”。
- LLM-Wiki 页暗示“热门资料 = 正确课程内容”。
- 低分只显示分数，没有回流对象。
- source_files 不可见。
- 用户无法说出事故、负责层和证据。
