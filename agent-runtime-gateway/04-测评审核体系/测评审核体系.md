# 测评审核体系

生成日期：2026-06-29  
用途：定义如何证明 Agent 系统准确、稳定、安全、可审计，而不是只看一次演示是否成功。

## 1. 测评对象

企业级 Agent 至少测 8 类对象：

| 对象 | 关键问题 | 示例指标 |
|---|---|---|
| 最终答案 | 回答是否正确、有证据、不编造 | answer accuracy、faithfulness |
| 工具调用 | 是否选对工具、参数是否正确 | tool call accuracy、schema pass rate |
| Agent 轨迹 | 是否按计划执行、是否重复、是否恢复 | step efficiency、loop rate、recovery success |
| 记忆系统 | 是否记对、忘对、召回对 | recall、pollution、staleness、privacy leakage |
| Gateway | 路由、限流、fallback、预算是否生效 | routing success、fallback success、cost per run |
| 安全治理 | 是否阻止越权、注入、泄漏、危险动作 | block rate、false refusal、secret leakage |
| 多 Agent | handoff 是否正确、冲突是否被仲裁 | handoff success、role violation、deadlock rate |
| 可观测性 | 失败是否能复盘 | trace completeness、span coverage、replay success |

## 2. 四层评测集

| 评测集 | 用途 | 何时运行 |
|---|---|---|
| Smoke Set | 快速冒烟，确认主链路没断 | 每次小改动 |
| Golden Set | 常规任务准确率和稳定性 | 每次 prompt/model/tool 变更 |
| Red Team Set | 注入、越权、泄漏、危险动作 | 每次发布前 |
| Regression Set | 历史故障复现 | 每次修 bug 后加入，发布前必跑 |

## 2.1 Design-only 审核

当前学习模式下，用户可以不写代码，但不能跳过审核。Design-only 阶段至少审核四类设计证据：

| 证据 | 审核问题 | 不合格信号 |
|---|---|---|
| 评测样例 | 是否覆盖正常、困难、越权、注入和回归 | 只有 happy path |
| Trace 设计 | 是否能定位失败发生在哪一层 | 只有最终答案，没有过程 |
| Audit 设计 | 是否记录主体、策略、工具、结果和错误 | 权限判断无证据 |
| 发布门禁 | 是否明确哪些失败阻塞上线 | 所有失败都只算普通低分 |

你不需要亲自实现 eval runner，但必须能判断：

- 这个 case 测的是答案、工具、轨迹、记忆、安全还是治理？
- 失败是否必须阻塞发布？
- 如果模型或 prompt 更新，这个 case 是否需要回归？
- 结果变差时，应该看 trace、audit、policy、memory 还是 retrieval？

## 2.2 RAG 评测集

RAG 评测集必须单独分层，因为它同时影响知识、权限、引用、新鲜度和成本。

| RAG 分层 | 要防止的失败 | 阻塞策略 |
|---|---|---|
| smoke | 基础链路断裂 | 关键业务链路阻塞 |
| hard_retrieval | 同义词、短查询、多轮省略召回失败 | 视业务关键度阻塞 |
| citation | 答案无证据或引用错误 | 通常阻塞 |
| freshness | 旧政策覆盖新政策 | 通常阻塞 |
| security | prompt injection、数据投毒 | 必须阻塞 |
| tenant_acl | 跨租户召回 | 必须阻塞 |
| memory_boundary | 记忆污染或越权召回 | 必须阻塞 |

## 3. Agent 轨迹评测

不要只看最后答案，还要看过程：

- 是否调用了不必要工具。
- 是否重复调用同一工具。
- 是否在工具失败后正确恢复。
- 是否把用户输入直接塞进工具参数。
- 是否越权访问资源。
- 是否产生不可审计的中间状态。

轨迹评测最小数据结构：

```text
run_id
task_id
expected_steps
actual_steps
tool_calls
policy_decisions
memory_reads
memory_writes
errors
final_result
verdict
```

## 4. 记忆系统评测

记忆必须单独评测，因为它会长期影响未来任务。

| 指标 | 说明 |
|---|---|
| memory_recall | 应该召回的记忆是否召回 |
| memory_precision | 召回内容是否相关 |
| memory_pollution | 错误事实是否被写入 |
| stale_memory_rate | 过期信息是否仍被使用 |
| privacy_leakage | 是否跨用户、跨租户、跨权限召回 |
| deletion_success | 删除/遗忘请求是否生效 |
| source_coverage | 每条长期记忆是否有来源 |

最低红队用例：

- 用户要求记住明显错误事实。
- 用户要求记住敏感信息。
- 用户尝试读取别人的记忆。
- 旧事实被新事实覆盖后，系统仍召回旧事实。
- prompt injection 诱导 Agent 把临时内容写入长期记忆。

## 5. Gateway 与工具治理评测

必须覆盖：

- 未认证请求被拒绝。
- 超预算请求被拒绝或降级。
- provider 失败后 fallback 生效。
- 只读工具允许。
- 写工具需要审批。
- 危险工具默认拒绝。
- 工具参数 schema 错误被拒绝。
- `operation_id` 重复不会重复执行副作用。

关键指标：

- routing_success_rate。
- fallback_success_rate。
- p95_latency。
- cost_per_successful_run。
- unauthorized_tool_block_rate。
- audit_event_coverage。

## 6. 多 Agent 协同评测

多 Agent 失败通常不是最终答案错，而是协作过程失控。

必须测：

- handoff 是否包含完整状态。
- reviewer 是否能发现 executor 的错误。
- verifier 是否真的运行验证。
- planner 是否过度拆任务。
- 多 agent 是否同时写同一资源。
- 冲突是否有仲裁。
- 最大轮数是否生效。
- 无进展时是否停下。

指标：

- handoff_success_rate。
- role_boundary_violation_rate。
- duplicate_work_rate。
- conflict_resolution_success。
- loop_guard_trigger_rate。
- human_escalation_precision。

## 7. 安全红队评测

红队必须覆盖：

- prompt injection。
- tool injection。
- 越权访问。
- secret 泄漏。
- 数据外传。
- 危险命令。
- 跨租户资源访问。
- 记忆污染。
- 审计绕过。

通过标准：

- critical case block rate = 100%。
- 所有拒绝都要有结构化错误码。
- 拒绝本身必须进入 audit event。
- false refusal 需要统计，不能为了安全把正常任务全拒绝。

## 8. 阶段门禁：Design-only 与 Implementation-later

任何阶段进入下一阶段前，都不得只用“看起来对”作为依据。但当前学习模式分两层：

| 模式 | 通过证据 | 不要求 |
|---|---|---|
| Design-only | 设计卡、评测样例、坏输入表、trace/audit 字段设计、发布阻塞判断、用户复述 | 不要求代码、运行命令、真实 run_id |
| Implementation-later | schema tests、smoke eval、failure-path test、trace/audit coverage check、eval report | 不允许只停留在文档 |

Design-only 阶段至少通过：

```text
contract / boundary responsibility review
smoke case design
one failure-path or attack case design
trace / audit field design
release blocker decision
teaching explanation and user restatement
```

Design-only 每次进入下一阶段，必须保存：

- 关联设计文档。
- 评测样例或坏输入样例 ID。
- 预期通过/失败判断。
- trace/audit 应记录的字段。
- 失败项是否应进入 regression set。
- 是否阻塞后续工程实现。

Implementation-later 或 MVP 发布前，才要求保存：

- 评测集版本。
- 运行命令。
- 通过/失败 case。
- trace_id 或 run_id。
- 失败项是否进入 regression set。
- 是否阻塞发布。

MVP-complete 发布前必须通过：

- golden set。
- red team set。
- memory eval。
- trajectory eval。
- gateway policy eval。
- sandbox adversarial probe。
- trace replay audit。

Phase 10/P1 进入 beta 前再追加：

- multi-agent handoff eval。
- resource lock/deadlock eval。
- arbiter conflict resolution eval。
- reviewer/verifier evidence eval。

## 8.1 工业级阻塞门禁

以下失败必须阻塞发布或阻塞进入下一阶段：

| 失败 | 阻塞原因 |
|---|---|
| critical red team 未拦截 | 安全边界不成立 |
| tool call 无 audit event | 无法审计 |
| policy decision 无记录 | 权限不可复盘 |
| trace 缺失 | 无法定位长线任务失败 |
| memory write 无来源 | 记忆污染不可控 |
| 跨租户记忆召回 | 隐私边界不成立 |
| 重复 operation_id 产生重复副作用 | 长线任务不可靠 |
| checkpoint 无法恢复 | 长线运行不成立 |
| 评测失败未进入 regression set | 回归体系不成立 |

## 8.2 三段式评测闭环

| 阶段 | 目的 | 证据 |
|---|---|---|
| CI 评测 | 阻止明显退化进入主分支或发布包 | test command、eval report、case diff |
| 线上观测 | 发现真实用户路径、成本、延迟、工具失败和安全拦截 | trace、audit、metrics、sample review |
| 回流评测集 | 把事故、误拒、漏拦、记忆污染变成 regression case | new regression id、owner、修复版本 |

不能只在本地跑一次 eval 就宣布可靠。长线 Agent 的准确率来自持续闭环：评测阻塞、运行观测、失败回流、再次阻塞。

## 10. 教学重点

你最终要学会的不是“跑一个 eval 工具”，而是建立判断体系：

- 质量问题看 eval。
- 安全问题看 red team。
- 过程问题看 trace。
- 记忆问题看 memory eval。
- 多 Agent 问题看 handoff 和共享状态。
- 上线问题看发布门禁和审计报告。
