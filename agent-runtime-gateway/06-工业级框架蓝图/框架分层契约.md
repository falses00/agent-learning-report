# 框架分层契约

生成日期：2026-06-29  
目标：规定每层负责什么、不负责什么，避免框架变成一团无法治理的代码。

## 1. 分层总览

```text
Application
Gateway
Runtime
Orchestration
Model
Tool
Memory
Evaluation
Observability
Security
Governance
Storage
```

## 2. Layer Contract

| 层 | 输入 | 输出 | 责任 | 不负责 |
|---|---|---|---|---|
| Application | 用户请求 | 标准请求 | 业务入口和展示 | 权限裁决 |
| Agent Gateway | request、principal | normalized request | 身份、限流、预算、路由 | 推理 |
| Runtime | normalized request、manifest | run/step/event | 状态机、恢复、错误 | 工具权限 |
| Orchestration | run state | next action | 单/多 Agent 控制流 | 直接执行工具或绕过 Tool Gateway |
| Model Gateway | model request | model response | provider 路由、fallback、成本 | 业务策略 |
| Tool Gateway | ToolCall | ToolResult 或拒绝 | schema、policy、凭据、审计 | 生成推理 |
| Memory | read/write request | memory refs/results | 写入门禁、召回、遗忘 | 替代事实验证 |
| Evaluation | run/trace/result | verdict/report | 质量和安全判断 | 运行时权限 |
| Observability | events/spans | trace/log/metric | 可观测、复盘、成本 | 业务决策 |
| Security | risk context | allow/deny/isolate | 策略、隔离、secret | 任务规划 |
| Governance | versions/policies/evals | release decision | 发布、审批、回滚 | 模型推理 |
| Storage | persisted records | durable data | 事务、查询、迁移 | 业务语义 |

## 3. 必须标准化的对象

- `AgentManifest`
- `Run`
- `Step`
- `Event`
- `ToolCall`
- `ToolResult`
- `PolicyDecisionInput`
- `PolicyDecision`
- `MemoryRecord`
- `EvalCase`
- `EvalResult`
- `AuditEvent`
- `TraceSpan`
- `ErrorModel`

## 4. 跨层规则

- 任意跨层调用必须传递 `trace_id`。
- 任意工具调用必须有 `operation_id`。
- 任意拒绝必须有 `ErrorModel`。
- 任意策略决策必须有 audit event。
- 任意长期记忆写入必须有来源和过期/撤销机制。
- 任意发布必须绑定版本集合。

## 5. 禁止跨层偷懒

禁止：

- Runtime 直接调用真实外部工具，绕过 Tool Gateway。
- Tool Gateway 从 prompt 中读取权限结论。
- Memory 直接写入未经审核的用户输入。
- Evaluation 只看最终答案，不看 trace。
- Observability 记录 secret。
- Governance 跳过 eval gate 发布。

## 6. 工业级代码审查问题

每次实现一个模块都问：

```text
它属于哪一层？
它的输入输出是否结构化？
它是否传递 trace_id？
它失败时是否有 ErrorModel？
它是否产生审计证据？
它是否有正常路径和失败路径测试？
它是否越权承担了别的层职责？
```
