# 工业级能力基线

生成日期：2026-06-30  
目标：定义本教学项目什么时候只能叫 demo，什么时候可以叫可实用 MVP，什么时候才接近工业级 Agent 框架。

## 1. 基本判断

工业级 Agent 不等于“能回答问题”，也不等于“能调用工具”。它至少要把不确定的模型行为放入可观测、可恢复、可评估、可审计、可隔离、可回滚的工程边界中。

本项目采用五级能力基线：

| 等级 | 名称 | 可对外说法 | 核心要求 |
|---|---|---|---|
| L0 | 教学切片 | 可演示链路 | 单条端到端路径跑通，有 trace 和 audit |
| L1 | 可实用 MVP | 内部试点 | 支持 checkpoint、policy、tool gateway、eval gate、rollback |
| L2 | 生产候选 | 小范围生产 | 有 SLO、告警、事故响应、数据治理、发布证据包 |
| L3 | 生产级 | 关键业务辅助 | 多租户隔离、权限审计、恢复演练、红队回归稳定通过 |
| L4 | 企业关键级 | 高风险业务需谨慎 | 合规审计、灾备、容量规划、变更治理、持续安全验证 |

## 2. 十一项硬能力

| 能力域 | L1 最低要求 | L2/L3 追加要求 | 不能接受的做法 |
|---|---|---|---|
| Agent Gateway | 统一入口、身份、tenant/workspace、trace_id | 配额、熔断、租户级策略、错误归一化 | 应用绕过网关直接调用工具 |
| Runtime | Run/Step/State/Checkpoint | resume/retry/fork/cancel 语义固定，支持恢复演练 | 只保存 chat history 当状态 |
| Tool Gateway | schema 校验、RBAC、audit、operation_id | risk level、审批、dry-run、结果脱敏、工具版本锁定 | Prompt 自己决定权限 |
| Model Gateway | provider adapter、fallback、成本记录 | 模型能力分级、质量回归门禁、预算策略 | fallback 后不记录退化 |
| Memory | session/artifact summary、写入门禁 | 语义记忆、过期、删除、污染纠正、跨租户隔离 | 自动把所有对话写入长期记忆 |
| Evaluation | golden/redteam/tool/security/long-run 基线 | CI gate、线上抽样、漂移监控、评测报告归档 | 只看最终回答，不看过程 |
| Observability | Run trace、model/tool/policy/memory span | OpenTelemetry 导出、成本仪表盘、trace replay | 失败后只能看文本日志猜原因 |
| Security | 租户隔离、secret 不进 prompt、最小权限 | DLP、egress control、sandbox profile、威胁模型 | 只靠提示词写“不要越权” |
| Multi-Agent | handoff schema、角色边界、loop guard | 资源锁、仲裁、死锁检测、独立 review/verifier | 多 Agent 自由聊天互相转述状态 |
| SRE | 指标、告警、runbook、回滚路径 | SLO、错误预算、容量和成本计划、故障演练 | 上线后没有恢复脚本和负责人 |
| Governance | 版本、审批、发布记录 | 变更证据包、审计导出、合规保留策略 | prompt/model/tool/policy 分开乱改 |

## 3. 教学阶段对应关系

| 教学阶段 | 必须形成的工业级能力 |
|---|---|
| Phase 1 Contract Layer | schema、错误模型、版本边界 |
| Phase 2 Minimal Runtime | Run/Step/State、结构化 trace |
| Phase 3 Gateway + Tool Governance | Tool Gateway、Policy Gateway、Audit |
| Phase 4 Long-Running Task | checkpoint、resume、幂等、人工审批 |
| Phase 5 MCP + Tool Ecosystem | MCP adapter、tool registry、credential broker |
| Phase 6 Memory System | memory lifecycle、隔离、污染测试 |
| Phase 7 Evaluation | golden/redteam/trajectory/security gate |
| Phase 8 Observability | OTel trace、failure analysis、cost |
| Phase 9 Sandbox | network/filesystem/process/secret isolation |
| Phase 10 Multi-Agent | handoff、shared state、loop/deadlock guard |
| Phase 11 Governance + Release | release gate、rollback、audit export |

## 4. 可实用 MVP 的最低承诺

如果只完成 L0，项目只能用于教学演示。  
如果完成 L1，项目可以用于低风险内部试点。  
如果要进入真实生产，至少要完成 L2，并经过生产就绪审核。

L1 必须同时满足：

- 一个真实的长线任务可通过 Gateway 进入 Runtime。
- 每个 Step 都有 checkpoint、trace、audit。
- 所有工具调用都经过 Tool Gateway 和 Policy Gateway。
- 高风险工具默认不能自动执行，必须 dry-run 或人工审批。
- 有 golden、redteam、tool-call、long-run recovery 四类评测。
- 发布时能一起锁定 agent/prompt/model/tool/policy/eval set 版本。
- 失败后能通过 run_id/trace_id 找到原因并回滚。

## 5. 工业级反思问题

每完成一个阶段，都必须问：

1. 这一步是否只是“能跑”，还是已经可恢复、可审计、可评估？
2. 失败后是否能定位到模型、工具、记忆、策略、沙箱或运行时哪一层？
3. 是否存在 prompt 绕过权限的路径？
4. 是否存在跨租户、跨工作区、跨用户的数据污染？
5. 是否有证据证明准确率没有退化，而不是凭感觉？
6. 是否有回滚路径和停止开关？

结论规则：只要以上任一问题没有证据，本阶段不能标记为工业级完成。
