# 工业级发布门禁矩阵

生成日期：2026-06-30  
目标：让 Agent 的每次变更都能证明“可以发布、可以回滚、可以审计”。

## 1. 发布不是只发代码

Agent 发布对象是一个组合包，不是单个服务版本。每次发布必须同时锁定：

- agent manifest version。
- prompt version。
- model/provider routing version。
- tool registry version。
- policy version。
- memory schema/retrieval config version。
- eval dataset version。
- sandbox profile version。
- runtime/gateway code version。
- migration version。

任一项变化，都可能改变行为，因此都要进入发布门禁。

## 2. 变更类型与最低门禁

| 变更类型 | 最低测试 | 必须审查 | 发布风险 |
|---|---|---|---|
| 文档变更 | link/path check | 内容是否误导教学路线 | 低 |
| Prompt 变更 | golden + redteam + trajectory eval | 是否扩大工具权限或隐藏失败 | 中 |
| Model/provider 变更 | golden + latency/cost + safety eval | fallback、成本、能力等级 | 中 |
| Tool schema 变更 | schema + contract + policy tests | 参数兼容、结果脱敏、operation_id | 高 |
| 新增工具 | contract + RBAC/ABAC + redteam + audit | 风险等级、审批、凭据、egress | 高 |
| Policy 变更 | allow/deny regression + bypass tests | false allow、false deny、审批路径 | 高 |
| Memory 策略变更 | memory write/read/leakage eval | 跨租户隔离、过期、删除、污染 | 高 |
| Runtime 状态机变更 | checkpoint/resume/retry/cancel tests | 幂等、终态一致性、恢复语义 | 高 |
| MCP server 变更 | compatibility + tool inventory + policy tests | 来源、版本、权限、健康检查 | 高 |
| Sandbox profile 变更 | escape/egress/resource tests | 网络、文件、进程、secret 暴露 | 高 |
| Evaluation set 变更 | dataset diff + judge calibration | 是否降低门槛或删除关键用例 | 中 |
| 数据迁移 | migration dry-run + rollback drill | 备份、回滚、数据完整性 | 高 |

## 3. 发布等级

| 等级 | 使用场景 | 门禁要求 |
|---|---|---|
| docs-only | 只改教学或说明文档 | 链接、路径、术语一致性检查 |
| experimental | 本地实验 | 单元/契约测试，标明不可生产 |
| beta | 内部试点 | contract、integration、eval、trace/audit 覆盖 |
| production-candidate | 小范围生产候选 | SLO、告警、回滚演练、红队 critical 通过 |
| production | 生产 | 发布证据包、审批、错误预算、事故响应值班 |
| high-risk-production | 涉及写操作、敏感数据或外部系统 | 双人审批、dry-run、分阶段放量、强回滚 |

## 5. 阻断条件

出现以下任一情况，禁止进入 beta 以上环境：

- critical redteam case 未通过。
- destructive tool 没有审批或 dry-run。
- tool call 没有 operation_id。
- policy decision 没有 audit event。
- trace 缺失或不能定位失败层。
- memory leakage 测试失败。
- sandbox egress/resource 限制缺失。
- 迁移不可回滚且没有备份。
- secret、token、private key 出现在 prompt、trace、log 或 memory。
- eval set 被降低门槛但没有审批记录。

## 6. 分阶段放量

生产发布建议采用：

1. shadow：只观察，不影响用户结果。
2. canary 1%：低风险租户或内部用户。
3. canary 10%：观察错误预算和成本。
4. guarded rollout：高风险工具继续人工审批。
5. full rollout：满足 SLO 且无 P0/P1 事故。

任何阶段触发阻断条件，必须回滚或暂停放量。

## 7. 教学检查点

学习者需要能回答：

- 为什么 Agent 发布要锁定 prompt/model/tool/policy/eval set，而不是只发代码？
- 为什么“评测通过率提升”仍然可能禁止发布？
- 如何证明一个高风险工具不会在重试时重复执行？
- 回滚时哪些状态可以保留，哪些状态必须重新生成？

不能回答这些问题，就不能进入 Phase 11 的治理与发布实现。
