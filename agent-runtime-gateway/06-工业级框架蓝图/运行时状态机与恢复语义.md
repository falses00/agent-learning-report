# 运行时状态机与恢复语义

生成日期：2026-06-30  
目标：把长线 Agent 任务从“对话循环”变成可恢复、可审计、可验证的 Runtime 协议。

## 1. 核心原则

长线任务不能只依赖上下文窗口。Runtime 必须把每一次执行拆成 Run 和 Step，并为每个 Step 记录输入、输出、状态、工具副作用、checkpoint 和审计事件。

基本原则：

- Run 是一次用户目标的执行实例。
- Step 是 Run 内最小可重试执行单元。
- Checkpoint 是恢复依据，不是聊天记录备份。
- operation_id 是副作用幂等依据。
- Policy decision 和 audit event 是不可省略的恢复上下文。

## 2. Run 状态

| 状态 | 含义 | 是否终态 | 允许动作 |
|---|---|---:|---|
| created | Run 已创建，未入队 | 否 | enqueue/cancel |
| queued | 等待执行资源 | 否 | start/cancel/expire |
| planning | 正在生成计划或选择 Agent | 否 | continue/pause/fail |
| running | 正在执行 Step | 否 | checkpoint/pause/cancel/fail |
| waiting_approval | 等待人工审批 | 否 | approve/reject/cancel/expire |
| waiting_external | 等待外部系统回调或资源 | 否 | resume/cancel/expire |
| checkpointed | 已持久化可恢复状态 | 否 | resume/fork/cancel |
| retrying | 按策略重试 | 否 | continue/fail/quarantine |
| paused | 用户或系统暂停 | 否 | resume/fork/cancel |
| quarantined | 发现安全、数据或一致性风险，隔离等待处理 | 否 | review/recover/cancel |
| succeeded | 成功完成 | 是 | export/replay |
| failed | 不可恢复失败 | 是 | inspect/retry_as_new |
| cancelled | 用户或系统取消 | 是 | inspect |
| expired | 超时过期 | 是 | inspect/retry_as_new |

## 3. Step 状态

| 状态 | 含义 | 可重试要求 |
|---|---|---|
| step_created | Step 已创建 | 尚未产生副作用，可安全重试 |
| model_calling | 正在调用模型 | 失败可重试，但要记录 prompt/model version |
| tool_intent_ready | 模型提出工具意图 | 必须进入 schema/policy 校验 |
| policy_checking | 正在做权限和风险决策 | 决策结果必须审计 |
| waiting_approval | 高风险动作等待审批 | 审批票据进入 checkpoint |
| tool_executing | 工具执行中 | 必须有 operation_id |
| tool_completed | 工具成功 | 结果经过 schema 与 redaction |
| checkpoint_written | checkpoint 成功写入 | 可恢复点 |
| step_failed_retryable | 可重试失败 | 进入 retrying 或 paused |
| step_failed_terminal | 不可重试失败 | Run failed 或 quarantined |

## 4. 状态转移规则

| 转移 | 触发条件 | 必要证据 |
|---|---|---|
| created -> queued | 请求通过入口校验 | request_id、trace_id、principal |
| queued -> planning | runtime 分配资源 | runtime_worker_id |
| planning -> running | 计划生成并通过 schema | plan_version、agent_version |
| running -> checkpointed | Step 成功并写 checkpoint | checkpoint_id、state_hash |
| running -> waiting_approval | 高风险工具或策略要求 | policy_decision_id、approval_ticket |
| waiting_approval -> running | 审批通过 | approver、approval_record |
| waiting_approval -> paused | 审批拒绝或等待超时 | rejection_reason/timeout |
| running -> retrying | 可恢复错误 | error_code、retry_policy |
| retrying -> running | 重试预算未耗尽 | retry_count、backoff |
| retrying -> quarantined | 重复失败或疑似安全风险 | failure_pattern、risk_reason |
| checkpointed -> running | resume | checkpoint_id、resume_reason |
| checkpointed -> paused | 用户暂停或预算不足 | pause_reason |
| paused -> running | 用户或系统恢复 | resume_actor、checkpoint_id |
| any active -> cancelled | 用户/管理员取消 | cancel_actor、cancel_reason |
| any active -> expired | TTL 到期 | ttl_policy |
| running/checkpointed -> succeeded | 完成目标且通过终态校验 | final_result_hash、eval_summary |

## 5. Checkpoint 内容

每个 checkpoint 至少包含：

```json
{
  "checkpoint_id": "ckp_...",
  "run_id": "run_...",
  "step_id": "step_...",
  "agent_version": "agent_v...",
  "prompt_version": "prompt_v...",
  "model_version": "provider/model",
  "tool_registry_version": "tools_v...",
  "policy_version": "policy_v...",
  "eval_set_version": "eval_v...",
  "state_hash": "sha256...",
  "state": {},
  "pending_approvals": [],
  "completed_operations": [],
  "memory_refs": [],
  "artifact_refs": [],
  "created_at": "ISO-8601"
}
```

禁止只保存自然语言总结作为 checkpoint。总结可以作为辅助，但恢复必须依赖结构化状态。

## 6. 恢复语义

| 动作 | 语义 | 成功标准 |
|---|---|---|
| resume | 从最近可信 checkpoint 继续 | 不重复已完成副作用 |
| retry_step | 只重试当前 Step | 同一 operation_id 不产生重复写入 |
| retry_as_new | 从终态失败 Run 创建新 Run | 新 run_id，引用原始 run_id |
| fork | 从 checkpoint 分叉试验新策略 | 新 run_id，新 trace，保留 parent_run_id |
| cancel | 停止未完成工作 | 不再调工具，保留审计和状态 |
| quarantine | 隔离可疑 Run | 阻止自动恢复，等待人工 review |

## 7. 幂等和副作用

所有写操作必须满足：

- 调用前生成 `operation_id`。
- Tool Gateway 记录 operation_id、tool_name、resource、args_hash。
- 重试时先查 operation_id 是否已经完成。
- 工具返回结果必须可关联到 operation_id。
- 对不支持幂等的外部系统，必须使用 dry-run、预提交或人工审批。

## 8. 教学练习

学习者必须能完成三组实验：

1. kill runtime 进程后，从 checkpoint 恢复，不重复工具写入。
2. 模型提出越权工具调用后，Run 进入 paused/quarantined，并写 audit。
3. 同一 checkpoint fork 两条路线，对比 trace、成本和 eval 结果。

通过标准：实验结果能用 run_id、step_id、checkpoint_id、operation_id 追溯，而不是靠截图或口头解释。
