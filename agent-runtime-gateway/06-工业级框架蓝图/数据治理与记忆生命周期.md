# 数据治理与记忆生命周期

更新日期：2026-07-13
目标：让记忆系统可控、可删、可审计、可评测。

方法原理、14 类方案比较、基准与趋势见：[Agent Memory 方法谱系与工业选型（2026）](Agent-Memory方法谱系与工业选型-2026.md)。本页专注生产治理契约。

## 1. 数据分类

| 分类 | 示例 | 默认策略 |
|---|---|---|
| Public | 公开文档 | 可检索 |
| Internal | 内部知识库 | 按 tenant/workspace 权限过滤 |
| Confidential | 客户数据、工单细节 | 默认不写长期记忆 |
| Secret | API key、token、密码 | 禁止进入 prompt、memory、trace |
| Regulated | 法务/医疗/金融敏感数据 | MVP 禁止 |

## 2. 记忆生命周期

```text
capture
-> select / write_gate
-> extract / normalize
-> store / index
-> consolidate / evolve
-> retrieve
-> context assembly
-> use / act
-> correct / expire / delete
-> evaluate / audit
```

每条长期记忆必须包含：

- 来源。
- 写入原因。
- 权限范围。
- TTL 或失效条件。
- 是否可删除。
- 是否被评测覆盖。
- 版本、有效时间和冲突状态。
- 原文、索引、摘要、图边和缓存之间的删除传播状态。

## 3. 写入门禁

禁止写入：

- secret。
- 明显错误事实。
- 未授权用户数据。
- 跨租户信息。
- prompt injection 内容。
- 临时任务噪音。

需要人工或规则确认：

- 用户偏好。
- 企业流程经验。
- 可复用处理策略。
- 经过工具验证的事实。

## 4. 记忆操作控制矩阵

| 操作 | 权限要求 | 必填证据 | 阻断条件 | 评测覆盖 |
|---|---|---|---|---|
| `memory.add` | principal 有写入当前 tenant/workspace 权限 | source、reason、classification、TTL、confidence | secret、跨租户、未验证高风险事实、prompt injection | pollution、privacy leakage |
| `memory.search` | principal 有读取目标 scope 权限 | query、tenant、workspace、role、filters | 过滤条件为空、请求跨租户、敏感级别不匹配 | recall、precision、privacy |
| `memory.update` | owner/reviewer 或策略授权 | previous_version、diff、reason、evidence | 无版本号、覆盖来源、降级分类 | correction、staleness |
| `memory.delete` | owner/admin 或合规请求 | delete_reason、requestor、scope | 无法定位 memory_id、删除后仍可召回 | deletion_success |
| `memory.expire` | 系统策略 | TTL、expired_at、policy_version | 过期记忆仍进入 context packer | stale_memory_rate |
| `memory.export` | 审计/合规授权 | audit_ticket、data_range、redaction_policy | 未脱敏、跨租户导出 | audit sampling |

任何记忆操作都必须写 audit event。长期记忆不能只靠向量库 ID 管理，必须有结构化 metadata。

## 5. 召回策略

召回必须过滤：

- tenant。
- workspace。
- principal role。
- resource scope。
- data classification。
- staleness。

## 6. 记忆评测

必须测：

- 应召回是否召回。
- 不该召回是否被过滤。
- 过期记忆是否不再使用。
- 删除请求是否生效。
- 错误事实是否纠正。
- prompt injection 是否污染长期记忆。

## 7. 审计要求

记忆写入、修改、删除、召回都要记录：

- `memory_id`
- `run_id`
- `principal_id`
- `tenant_id`
- `source`
- `decision`
- `reason`
- `created_at`

## 8. 记忆供应商采用边界

Mem0、Letta、Zep、Graphiti 等项目可帮助理解记忆产品形态，但本项目不能默认采用“自动写入一切”的策略。

采用前必须确认：

- 是否支持 tenant/workspace/principal 过滤。
- 是否支持删除、纠错、TTL。
- 是否支持来源和置信度 metadata。
- 是否能导出审计事件。
- 是否能被 memory eval 覆盖。

## 9. 成本与性能边界

不要只统计回答 token。记忆系统至少拆分记录：

- 写入抽取与规范化成本。
- embedding、图更新、摘要合并和索引维护成本。
- 检索、重排、上下文组装和生成延迟。
- 删除传播、重建索引和冷启动时间。
- 每租户存储增长、无效召回率和过期记忆命中率。

先用会话隔离、结构化记录和受控检索建立基线；只有当长时依赖、关系查询或跨任务技能复用的评测确实失败时，才升级到层级、时序图或程序性记忆。
