# 工程实施架构

生成日期：2026-06-29  
目标：回答“进入 Implementation-later 代码阶段后，先建什么、怎么拆、怎么跑”。

适用范围：仅用于 **Implementation-later / 代码阶段**。当前默认学习模式仍是 Design-only，先掌握设计、评审、失败推演和验收门禁，不要求用户写代码或记代码。

## 1. MVP-practical 定位

**MVP-practical = 单租户工单处理 Agent 的可部署内测版。**

它不是完整企业平台，也不是开源通用框架。它是一个能在本地或单机环境运行的内部试点系统。

## 2. 推荐技术栈

默认推荐先用 Python：

| 层 | 推荐 | 原因 |
|---|---|---|
| API | FastAPI | 轻量、文档友好、适合教学 |
| Schema | Pydantic | 契约层直观，适合坏输入测试 |
| Runtime | Python service | Phase 2 可快速实现 Run/Step loop |
| 存储 | SQLite -> PostgreSQL | 教学先轻，试点可迁移 |
| 测试 | pytest | 简单、可 CI、适合 fixtures |
| 评测 | pytest fixtures 起步，后续 Promptfoo | 先控制复杂度 |
| 观测 | JSON trace 起步，后续 OpenTelemetry | 先可复盘，再接生态 |

如果用户明确选择 TypeScript，可改用 Node/Fastify/Zod/Vitest，但默认不要同时做双语言。

## 4. 服务边界

MVP-practical 先做单进程模块化服务：

```text
FastAPI app
-> Request Gateway
-> Runtime Runner
-> Tool Gateway
-> Policy Engine
-> Mock/File Tools
-> Audit/Trace Store
```

暂时不拆微服务。拆服务会提前引入网络、部署、权限、数据一致性和运维复杂度。

## 5. API 起步范围

| API | 用途 |
|---|---|
| `POST /runs` | 创建一次工单处理 run |
| `GET /runs/{run_id}` | 查询 run 状态 |
| `POST /runs/{run_id}/resume` | 恢复中断任务 |
| `GET /runs/{run_id}/trace` | 查看 trace |
| `GET /runs/{run_id}/audit` | 查看 audit events |
| `POST /evals/run` | 跑本地评测集 |

Phase 1 不实现 API，只定义 schema。Phase 2 起实现最小 API 或 CLI。

## 6. CLI 起步范围

CLI 比 UI 更适合教学：

```text
agent-demo run examples/ticket_001.json
agent-demo trace <run_id>
agent-demo eval golden
agent-demo replay <run_id>
```

UI Console 放到 Phase 11，不进入 MVP-practical。
