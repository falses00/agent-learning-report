# 安全与密钥管理

生成日期：2026-06-29  
目标：定义 MVP-practical 的最低安全边界。

## 1. 安全原则

- prompt 不是安全边界。
- 模型不能直接拿密钥。
- 工具调用必须先 schema 校验，再 policy 判断。
- 写操作默认拒绝或进入审批。
- 审计日志不能记录明文密钥。
- 真实生产凭据不进入教学阶段。

## 2. 身份与权限

MVP-practical 先用 mock principal：

- `principal_id`
- `tenant_id`
- `workspace_id`
- `roles`

保留未来接 OIDC/IAM 的接口，但不在第一版接真实身份系统。

## 3. 密钥管理

开发期：

- `.env.local` 本地保存。
- 不提交 Git。
- 不写进 prompt。
- 不写进 trace/audit。

试点期：

- 使用环境变量或 secret manager。
- Credential Broker 统一读取。
- 工具只拿临时凭据或代理调用能力。

## 4. 工具风险分级

| 风险 | 示例 | 默认策略 |
|---|---|---|
| 低 | 只读知识库查询 | 允许，但审计 |
| 中 | 查询内部状态 | 需要身份和资源校验 |
| 高 | 写工单、发邮件、改数据库 | 默认审批或 dry-run |
| 极高 | shell、浏览器、外网发送数据 | MVP 禁止 |

## 5. 必测安全用例

- prompt injection 诱导调用写工具。
- 用户伪造 `principal_id`。
- 用户传入跨租户 `resource_id`。
- 工具参数包含脚本或危险 payload。
- 模型试图读取环境变量。
- 错误日志包含敏感信息。

## 6. 最低通过标准

- critical 越权用例拦截率 100%。
- secret 不出现在 prompt、trace、audit、eval report。
- 写工具没有审批票据不得执行。
- 所有拒绝都有结构化 `ErrorModel` 和 audit event。
