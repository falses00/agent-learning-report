# 试点上线门禁

生成日期：2026-06-29  
用途：定义从教学 demo 进入内部试点前必须满足的条件。

## 1. 为什么需要门禁

Agent demo 能跑，不代表能给真实用户用。

进入内部试点前，必须证明：

- 它知道自己能做什么、不能做什么。
- 它失败时能被复盘。
- 它不会直接造成高风险副作用。
- 它的质量能被评测。
- 它的权限、工具和记忆有边界。

## 2. 试点前硬门禁

| 门禁 | 必须满足 |
|---|---|
| 场景边界 | 只处理低风险工单 |
| 数据边界 | 只接入脱敏或低敏只读数据 |
| 工具边界 | 只读工具默认允许，写工具默认拒绝或审批 |
| 身份边界 | 每个请求都有 principal、tenant/workspace |
| 审计边界 | 工具调用、策略决策、记忆写入必须有 audit event |
| 评测边界 | golden/red team/memory/trajectory eval 通过 |
| 观测边界 | 每个 run 都有 trace_id，失败可 replay |
| 安全边界 | secret 不进入 prompt，高风险工具不裸执行 |
| 人工边界 | Agent 输出为建议，关键动作需要人确认 |

## 3. 禁止进入试点的情况

出现以下情况，不能试点：

- 还没有评测集。
- 工具调用没有审计。
- 写操作可以被 prompt 绕过。
- 记忆可以跨用户或跨租户召回。
- 没有 trace，失败无法复盘。
- 真实凭据进入 prompt 或日志。
- 模型输出直接进入 shell、浏览器、数据库写操作。

## 5. 试点成功后才能扩展

试点成功后，才考虑：

- 接入真实 OIDC/IAM。
- 增加写操作审批。
- 增加真实知识库。
- 增加长期记忆。
- 增加 dashboard。
- 增加多 Agent 协同。
- 增加代码执行或浏览器自动化工具。
