# 数据模型与迁移计划

生成日期：2026-06-29  
目标：定义 MVP-practical 需要的最小数据模型和迁移策略。

## 1. 最小表

| 表 | 用途 |
|---|---|
| `agents` | Agent 基础身份 |
| `agent_versions` | Agent manifest 版本 |
| `runs` | 一次任务执行 |
| `steps` | Run 中的步骤 |
| `checkpoints` | 可恢复状态快照 |
| `tool_registry` | 工具定义、schema、风险等级 |
| `tool_calls` | 工具调用申请和执行状态 |
| `policy_decisions` | 策略判断结果 |
| `audit_events` | 审计事件 |
| `memory_records` | 记忆记录 |
| `eval_cases` | 评测用例 |
| `eval_results` | 评测结果 |

## 3. 数据保留

MVP-practical 默认：

- audit events 保留 90 天。
- trace 保留 30 天。
- eval results 长期保留。
- memory records 必须支持删除或失效标记。

真实生产前必须重新定义保留策略和合规要求。

## 4. 迁移策略

Phase 1-2：

- 可以先用 JSON 文件或 SQLite。
- schema 稳定后再建表。

Phase 3-4：

- 引入 SQLite/PostgreSQL 迁移。
- 每次 schema 变化必须有迁移脚本。

prod-lite：

- 使用 PostgreSQL。
- 迁移命令可重复运行。
- 迁移前有备份策略。

## 5. 种子数据

必须提供：

- 一个工单 Agent manifest。
- 两个工具：`ticket.get`、`kb.search`。
- 一组 mock 工单。
- 一组知识库片段。
- 一组 golden eval cases。
- 一组 red team eval cases。
