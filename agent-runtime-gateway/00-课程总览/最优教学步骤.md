# 最优教学步骤

生成日期：2026-06-29  
用途：作为本项目后续教学的主路线。每一阶段都要产出工程结果、教学说明、验证证据和反思结论。

当前默认采用 `Design-only`：你不写代码，也不记代码。每阶段先训练设计判断、架构评审、失败推演和验收复述；工程实现作为后续可选承接。

## 1. 总体学习循环

```text
需求澄清
-> subagent 增强提示词
-> 最新资料核验
-> 设计最小可评审产物
-> 失败路径推演
-> 架构评审
-> 验收
-> 审核
-> 教学复盘
-> 进入下一阶段或停下修正
```

每次任务必须先问：

- 本阶段解决哪个失控风险？
- 有没有参考当前 GitHub/官方资料？
- 正常路径应该如何证明？
- 失败路径应该如何推演或测试？
- 设计是否有可审计、可评测、可回滚的证据？
- 结果如何教会用户复现？
- 是否达到 `06-工业级框架蓝图/工业级能力基线.md` 中对应等级的证据要求？

## 2. 12 个教学阶段 + 半懂起步桥

| 阶段 | 名称 | 做什么 | 测什么 | 学会什么 |
|---|---|---|---|---|
| Phase 0 | 课程总览与路线审核 | PRD、路线、调研、术语、验收清单 | 文档链接、路线一致性、资料新鲜度 | 知道为什么不是直接写 demo |
| Phase 0.1 | 半懂起步 | 用工单退款故事解释工业级 Agent | 三个过关问题、复述评分 | 知道建议和执行必须分开 |
| Phase 1 | 契约层 | AgentManifest、ToolCall、PolicyDecisionInput、ErrorModel | schema 正常/坏输入 | 用结构化契约约束 Agent |
| Phase 2 | 最小运行时 | message loop、Run/Step、mock model/tool、trace_id | 一次完整 run、工具失败、错误结构化 | Agent loop 如何运行 |
| Phase 3 | Agent 网关与工具治理 | Request Gateway、Model Gateway、Tool Gateway、Policy、Audit | 只读允许、写操作拒绝、限流/预算 | Prompt 不能决定权限 |
| Phase 4 | 长线任务与断点恢复 | checkpoint、resume、retry、cancel、HITL | 中断恢复、幂等、人工审批 | 长线任务如何可靠 |
| Phase 5 | MCP 与工具生态 | MCP adapter、Tool Registry、凭据代理、工具生命周期 | 工具注册、schema 版本、凭据隔离 | 协议和治理的边界 |
| Phase 6 | 记忆系统 | short/long/episodic/semantic memory、写入门禁、遗忘 | 命中率、污染率、跨租户泄漏、过期清理 | 记忆不是简单向量库 |
| Phase 7 | 测评审核与红队 | golden set、red team set、trajectory eval、CI gate | 准确率、工具选择、安全、记忆、回归 | 如何证明质量 |
| Phase 8 | 可观测性与审计 | trace/span、成本、延迟、replay、audit export | trace 完整性、失败定位、成本归因 | 如何复盘生产问题 |
| Phase 9 | 安全隔离与沙箱 | sandbox profile、网络/文件/进程隔离、secret 隔离 | 越权、逃逸、危险命令、数据外传 | 高风险执行如何隔离 |
| Phase 10 | 多智能体协同 | planner/executor/reviewer/verifier、handoff、共享状态、仲裁 | 死循环、重复工具、冲突、锁超时 | 多 Agent 如何不失控 |
| Phase 11 | 治理控制台与发布 | policy editor、版本、审批、回滚、报表 | 发布门禁、审计查询、回滚 | 企业如何上线和运营 |

## 3. 每阶段固定交付物

每个阶段至少交付：

- 一个核心设计产物：设计决策卡、契约、边界图或门禁表。
- 一个正常路径证明：测试、样例、流程图或清单验收。
- 一个失败路径或攻击路径推演。
- 一组架构评审问题。
- 一个用户复述验收。
- 一个教学说明。
- 一个阶段复盘。
- 一个下一阶段进入条件。
- 一个工业级证据项：trace、audit、eval、checkpoint、policy decision、release gate 或 recovery drill 中至少一种。

## 4. 学习者检查句式

完成每个阶段后，你要能说：

```text
这个阶段解决了 [风险]。
我设计了 [产物]。
它参考了 [项目/协议]，但没有照搬 [风险点]。
我用 [测试/命令/trace/评审清单/失败样例] 证明它有效。
它失败时我会查看 [日志/trace/评估报告]。
下一阶段要解决 [新风险]。
```

## 5. 不能跳过的验证

- 契约层必须测坏输入。
- Gateway 必须测越权。
- Runtime 必须测恢复。
- Memory 必须测污染和跨租户泄漏。
- Eval 必须测回归。
- Observability 必须测 trace 完整性。
- Sandbox 必须测危险操作被拒绝。
- Multi-Agent 必须测 handoff 失败和循环拦截。

这些验证不只是“质量保证”，也是教学本身。

在 Design-only 模式下，这些验证可以先表现为：坏输入表、失败场景卡、评测样例、trace 设计、审计字段和发布门禁。进入 Implementation-later 后，再把它们变成代码、测试和 CI gate。

## 6. RAG 问题驱动学习法

RAG 教学不从“选哪个框架”开始，而从失败模式开始：

| 失败模式 | 先学什么 | 再学什么 |
|---|---|---|
| 搜不到 | chunking、HyDE、hybrid search | GraphRAG/LightRAG |
| 搜到但噪声多 | rerank、MMR、metadata filter | late interaction |
| 答案没证据 | citation、faithfulness eval | Self-RAG/CRAG |
| 多跳失败 | query decomposition | GraphRAG/RAPTOR |
| 越权召回 | ACL、tenant namespace | multi-tenant RAG |
| 成本/延迟高 | cache、routing、topN 控制 | long-context route |
