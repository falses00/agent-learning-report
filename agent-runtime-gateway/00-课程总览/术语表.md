# 术语表

生成日期：2026-06-28  
适用项目：Agent Runtime Gateway  
用途：用统一中文解释核心概念，避免后续设计时同一个词有多种理解。

## 可视化界面中英显示规则

学习可视化前端默认采用中文优先双语：中文帮助理解，英文帮助对接官方资料、GitHub 项目、schema 和后续代码。

| UI 显示名 | English canonical term | 首次出现解释 |
|---|---|---|
| 智能体 | Agent | 不是聊天框，而是能按规则执行长线任务的系统 |
| 运行时 | Runtime | 管理 Run、Step、状态、恢复和错误的执行层 |
| 网关 | Gateway | 统称：请求、模型、工具进出的治理入口；界面必须说明是哪一种具体网关 |
| 请求网关 | Request Gateway | 请求进入系统时做身份、租户、限流和路由的入口 |
| 模型网关 | Model Gateway | 控制模型调用、fallback、预算和成本的层 |
| 工具网关 | Tool Gateway | 工具真正执行前做 schema、权限、凭据和审计检查的层 |
| 编排器 | Orchestrator | 决定单智能体、多智能体、人工审批和恢复流程 |
| 断点快照 | Checkpoint | 长线任务中断后可恢复的结构化快照 |
| 状态 | State | 当前任务的结构化进度、审批、工具结果和错误 |
| 策略 | Policy | 判断 allow、deny、approval 的规则层 |
| 检索增强 | RAG | 从受控知识源取证据，不等于默认可信事实 |
| 记忆 | Memory | 可写入、可撤销、可过期、可审计的长期信息 |
| 工具申请 | ToolCall | 模型提出的工具调用申请，不等于已经执行 |
| MCP | Model Context Protocol | 工具、资源、提示词接入协议，不等于企业可信工具治理 |
| 工具注册表 | Tool Registry | 记录工具名称、版本、schema、owner、权限和风险等级 |
| 凭据代理 | Credential Broker | 由平台代管密钥，Agent 不直接接触真实凭据 |
| 轨迹 | Trace | 用于定位问题的运行链路 |
| 追踪步骤 | Span | Trace 中的一次模型、工具、策略或 RAG 步骤 |
| 审计 | Audit | 用于证明谁在何时为何做了什么 |
| 审计事件 | Audit Event | 安全、合规和事故复盘需要保留的关键事实 |
| 评测 | Eval | 判断质量、安全和回归是否过门禁 |
| 评估门禁 | Eval Gate | 评测结果是否允许进入下一阶段的阻塞判断 |
| 发布门禁 | Release Gate | 判断版本是否允许进入下一阶段 |
| 沙箱 | Sandbox | 限制文件、网络、进程、环境变量和工具副作用的隔离环境 |
| 人工介入 | Human-in-the-loop | 高风险动作暂停，等待人审批、修改或拒绝 |
| 治理 | Governance | 版本、权限、评估、审计、发布和回滚的管理闭环 |
| 复述 | Restatement | 用户用自己的话证明真的理解 |

界面规则：

- 导航、按钮、复述题使用中文。
- 架构节点使用“中文 + English canonical term”，例如“策略 Policy”。
- JSON 字段、schema 名、外部项目名保留英文。
- 如果生成图或原型主要是英文，必须回到视觉方向阶段重做中文化。

## Agent 与运行

| 术语 | 白话解释 | 在本项目中的边界 |
|---|---|---|
| Agent | 能围绕目标进行多步推理、调用工具、保存状态并产出结果的程序 | 不等于单轮聊天机器人 |
| Agent Runtime | 执行 Agent 的运行时 | 负责 run、step、state、checkpoint、resume |
| Agent Gateway | Agent 请求进入平台的统一入口 | 负责认证、租户、限流、路由、审计入口 |
| Model Gateway | 模型调用网关 | 负责模型路由、fallback、成本、速率限制 |
| Orchestrator | 编排器 | 决定单 Agent、多 Agent、人工审批和恢复流程 |
| Checkpoint | 断点快照 | 长线任务中断后从这里恢复，而不是重新猜状态 |
| State | 结构化任务状态 | 包含目标、步骤、工具结果、审批状态，不只是 chat history |

## 工具与协议

| 术语 | 白话解释 | 在本项目中的边界 |
|---|---|---|
| ToolCall | Agent 申请调用工具的结构化请求 | 必须 schema 校验、权限判断、审计 |
| Tool Gateway | 工具调用网关 | 在工具真正执行前做校验、策略、凭据和结果过滤 |
| MCP | Model Context Protocol | 用于工具、资源、提示词的标准化接入协议 |
| Tool Registry | 工具注册表 | 记录工具名称、版本、schema、owner、权限、风险等级 |
| Credential Broker | 凭据代理 | Agent 不直接拿密钥，由平台代管和按策略下发 |
| Idempotency | 幂等 | 重试同一操作不会产生重复副作用 |

## 记忆与知识

| 术语 | 白话解释 | 在本项目中的边界 |
|---|---|---|
| Short-term Memory | 短期上下文 | 当前任务内可用，任务结束后可丢弃 |
| Long-term Memory | 长期记忆 | 跨任务复用，必须有来源、权限、过期和删除机制 |
| Episodic Memory | 事件记忆 | 记录某次任务发生过什么，适合审计和经验复盘 |
| Semantic Memory | 语义记忆 | 抽取后的事实或偏好，必须防止错误写入 |
| Memory Write Gate | 记忆写入门禁 | 判断哪些内容允许进入长期记忆 |
| Memory Retrieval Policy | 记忆召回策略 | 判断谁能召回什么记忆，以及在什么场景召回 |
| LLM-Wiki | LLM 维护的结构化知识层 | 本项目只采用知识治理模式，不复制外部仓库内容 |
| KnowledgeSource | 知识来源 | 记录外部项目、文档、课程或内部文件的 URL、license、更新时间和风险 |
| KnowledgeTopic | 知识主题 | 把知识映射到 Phase、系统层、RAG 问题和学习难度 |
| KnowledgeCard | 知识卡 | 可复述、可审核、可回流的中文学习单元，不等于事实本身 |
| KnowledgeVersion | 知识版本 | 记录知识卡什么时候变化、为什么变化、替代了哪个版本 |
| SourceReview | 来源审核 | 判断资料是否能进入课程，检查 license、freshness、质量、版权和翻译风险 |
| ImportQueue | 导入队列 | 收集新资料、低分反馈、RAG 失败和评测回归，等待审核后再更新课程 |
| Low-score feedback loop | 低分回流 | 学不会时不只记录分数，而是回流到 Concept、FailureCase、EvalCase、KnowledgeCard 或 RestatementCard |

## 评估与观测

| 术语 | 白话解释 | 在本项目中的边界 |
|---|---|---|
| Golden Dataset | 标准评估集 | 用来判断常规能力是否退化 |
| Red Team Dataset | 红队评估集 | 用来检测越权、注入、泄漏和危险行为 |
| Eval Gate | 评估门禁 | 代码、prompt 或模型变化前后必须通过的质量检查 |
| Trace | 链路追踪 | 一次请求从入口到模型、工具、策略、结果的完整轨迹 |
| Span | 追踪中的一个步骤 | 例如一次模型调用、一次工具调用、一次策略决策 |
| Audit Event | 审计事件 | 安全、合规和事故复盘需要保留的关键事实 |

## 安全与治理

| 术语 | 白话解释 | 在本项目中的边界 |
|---|---|---|
| Policy | 策略 | 用代码或配置表达“谁在什么条件下能做什么” |
| RBAC | 基于角色的权限控制 | 例如管理员、开发者、审计员 |
| ABAC | 基于属性的权限控制 | 根据租户、风险等级、资源标签、时间等属性判断权限 |
| Sandbox | 沙箱 | 限制代码、浏览器、shell、网络和文件系统的执行环境 |
| Human-in-the-loop | 人工介入 | 高风险动作暂停，等待人审批、修改或拒绝 |
| Governance | 治理 | 版本、权限、评估、审计、发布和回滚的管理闭环 |

## 项目阶段词

| 术语 | 白话解释 |
|---|---|
| MVP-slice | Phase 3 结束时的最小端到端演示，证明 Runtime + Gateway + Tool + Policy + Audit 链路可行 |
| MVP-complete | Phase 9 结束时 P0 能力基本打通，包含 runtime、gateway、checkpoint、MCP、memory、eval、observability、sandbox |
| P0 | MVP-complete 必须实现的能力 |
| P1 | Beta 阶段增强企业可用性的能力 |
| P2 | GA 或长期增强能力 |
| pattern only | 只学习设计模式，不直接引入依赖 |
| dependency allowed | 可作为候选依赖继续评估 |
| adapter candidate after license review | 许可证/商业条款复核后，才可作为适配器候选 |
