# RAG 路线反思-2026-06-30

生成日期：2026-06-30  
目标：反思这条 RAG 教学线是否真的能帮助学习者理解工业级、可长线运行的 Agent。

## 第一轮：不能从框架开始

结论：如果一开始教 LangChain、LlamaIndex、GraphRAG API，学习者容易误以为“会调用框架就是会做 RAG”。

正确路线应该是：

```text
先复现失败
-> 再定位层次
-> 再选择最小优化
-> 再做指标对比
-> 最后决定是否采用框架或小项目方案
```

这能训练判断力，而不是训练 API 记忆。

## 第二轮：大项目和小项目必须结合

大项目能教抽象：

- workflow。
- retriever。
- vector store。
- eval。
- observability。

小项目能教手感：

- chunking 为什么会丢上下文。
- HyDE 为什么能补语义缺口，也为什么会漂移。
- reranker 为什么能降噪，也为什么会增加延迟。
- CRAG 为什么能修复低质量检索，也为什么会增加成本。
- semantic cache 为什么能省钱，也为什么会误命中。

结论：大项目是地图，小项目是手术刀。

## 第三轮：RAG 必须和长线 Agent 一起学

普通 RAG 是一次性问答。工业级 Agent 的 RAG 会进入长线任务：

- 同一个 run 可能多次检索。
- checkpoint 要保存 context refs。
- index version 改变会影响 resume。
- 权限变化后要重新 policy check。
- 记忆写入会影响未来任务。
- 失败样本要进入 regression。

结论：RAG 不能只归在“知识库模块”，它必须和 Runtime、Gateway、Memory、Eval、Observability、Release Gate 连接。

## 第四轮：没有评测就没有优化

RAG 优化经常出现假进步：

- topK 变大，recall 提升但噪声更多。
- reranker 变准，但 p95 延迟不可接受。
- GraphRAG 多跳变好，但成本爆炸。
- HyDE 召回变好，但引入虚构方向。
- semantic cache 降成本，但把旧答案误命中。

结论：每个优化都必须有 baseline、消融、指标、失败样本和上线判断。

## 第五轮：安全和权限不能后补

RAG 的风险来自数据本身：

- 文档里可能有 prompt injection。
- 文档可能被投毒。
- 向量近邻可能跨租户。
- 生成后过滤已经太晚。
- citation 可能泄露未授权 doc_id。

结论：工业级 RAG 必须从检索前 ACL、tenant namespace、post-check、red team、audit 开始设计。

## 第六轮：这条路线能不能帮助真正理解

能，但前提是执行时不跳过实验。

必须坚持：

- 每一章先给失败案例。
- 每一章只优化一个主要变量。
- Design-only 阶段每一章都要设计指标、阈值和上线判断；Implementation-later 阶段再实际跑指标。
- 每一章都要写“为什么不直接上更复杂方案”。
- 每一章都要说明如何接入 Agent Runtime Gateway。

如果只是读文档，不做失败样例、实验设计和验收判断，这条路线仍然会变成概念清单。真实指标运行属于后续 Implementation-later，但当前必须先把实验怎么判定设计清楚。

## 当前判断

当前 RAG 教学路线已经具备：

- 最新资料边界。
- 大项目基线。
- 小项目优化实验。
- 问题诊断矩阵。
- 安全、多租户、评测、观测。
- 长线 Agent 集成方式。

仍未完成：

- Implementation-later 的真实代码实验。
- Implementation-later 的本地评测集实际 runner。
- Implementation-later 的 baseline 与优化版实际运行指标。
- Implementation-later 的 RAG tool schema 和 Runtime 集成实现。

因此当前可说：

```text
RAG 教学路线已具备工业级设计骨架，可以进入实验设计；但还不能说已经具备可运行 RAG 系统。
```
