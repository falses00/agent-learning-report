# 学习路线图

生成日期：2026-06-28  
最近更新：2026-06-30  
适用项目：Agent Runtime Gateway  
用途：把完整 Agent 构建过程拆成可学习、可实施、可验证的阶段。

## 1. 学习方式

当前默认学习方式是 `Design-only`：

- 你不写代码，也不记代码。
- 你通过提问、追问、评审、复述来掌握设计。
- Codex 可以生成文档、契约、评测集、架构图和验收清单。
- 真正进入代码实现前，必须先证明你能判断设计是否工业级。

如果你是半懂学习者，先读：

1. [开始这里-半懂版](开始这里-半懂版.md)：今天只做哪三件事。
2. [Phase-00.1-半懂起步教学手册](../02-阶段教学手册/Phase-00.1-半懂起步教学手册.md)：先用工单故事建立直觉。
3. [每节课互动模板](每节课互动模板.md)：知道怎么提问、复述和卡住时求助。

建立第一张直觉图后，再读三个总控文档：

1. [课程地图](课程地图.md)：告诉你第一天和每个 Phase 该读什么。
2. [能力地图](能力地图.md)：告诉你每个工业能力在哪一阶段学、用什么练习证明。
3. [缺口审计](缺口审计.md)：告诉你当前哪些已完成、哪些还不能假装完成。

每个阶段都按同一个节奏推进：

```text
先明确目标
-> 调用 subagent 增强提示词
-> 检查资料是否最新
-> 设计最小可评审产物
-> 做失败路径推演
-> 做架构评审
-> 运行验证或清单验收
-> 教学复盘
-> 判断是否进入下一阶段
```

你每次都应该拿到几类产物：

- **设计产物**：设计决策卡、失败场景卡、架构评审问题、用户复述验收。
- **Design-only 产物**：业务故事、合法/坏输入样例、清单验收、复述评分。当前不要求写代码。
- **Implementation-later 产物**：代码、测试、CI、评估集或运行报告。后续进入工程阶段才需要。
- **教学产物**：这一阶段在 Agent 系统里解决什么问题，为什么不能省。
- **验收证据**：示例输入输出、风险清单、失败样例、复述评分或 reviewer 结论。

## 2. 总阶段路线

| 阶段 | 中文名称 | 主要问题 | 阶段产物 | 你应该学会 |
|---|---|---|---|---|
| Phase 0 | 课程总览与路线审核 | 到底要构建什么 Agent 平台，路线是否先进 | PRD、调研、追踪矩阵、教学循环、路线反思 | 能说清产品边界、参考依据和路线取舍 |
| Phase 0.1 | 半懂起步 | 工业级 Agent 和聊天 demo 有什么区别 | 工单故事复述卡、坏设计卡 | 能解释建议和执行为什么必须分开 |
| Phase 1 | 契约层 | Agent、工具、策略、错误如何结构化表达 | AgentManifest、ToolCall、PolicyDecisionInput、ErrorModel | 明白为什么 schema 是治理起点 |
| Phase 2 | 最小运行时 | Agent loop 如何运行 | Run、Step、mock model、mock tool、trace_id | 能解释 Agent 怎么一步步执行 |
| Phase 3 | Agent 网关与工具治理 | 请求如何进来，工具如何被管住 | Request Gateway、Model Gateway、Tool Gateway、Policy、Audit | 能跑通一次受控工具调用 |
| Phase 4 | 长线任务与断点恢复 | 长线任务如何暂停、恢复、重试 | checkpoint、resume、retry、cancel、HITL | 能解释任务状态不等于聊天记录 |
| Phase 5 | MCP 与工具生态 | 外部工具如何标准化接入 | MCP adapter、tool registry、凭据代理 | 能区分工具协议和企业治理 |
| Phase 6 | 记忆系统 | 什么能记、何时忘、谁能看、怎么测 | memory store、write gate、retrieval policy、memory eval | 能防止记忆污染和越权召回 |
| Phase 7 | 测评审核与红队 | 如何证明准确率、安全和轨迹没有退化 | golden set、red team set、trajectory eval、CI gate | 能设计可回归的 Agent 测评 |
| Phase 8 | 可观测性与审计 | 失败、成本、延迟和链路如何追踪 | trace、span、metrics、replay、audit export | 能从 trace 定位一次失败 |
| Phase 9 | 安全隔离与沙箱 | 高风险工具如何隔离执行 | sandbox profile、network/filesystem policy、secret isolation | 能按风险选择隔离等级 |
| Phase 10 | 多智能体协同 | 多角色如何分工而不失控 | handoff schema、shared state、resource lock、arbiter | 能控制协同边界和停止条件 |
| Phase 11 | 治理控制台与发布 | 企业如何管理版本、权限、审计和回滚 | admin console、policy editor、release gate、rollback | 能描述上线治理闭环 |

## 3. 每阶段固定提问模板

```text
继续 Phase X：阶段中文名。

请先调用合适的 subagent：
- planner：增强任务提示词与阶段计划
- researcher：如果涉及当前资料、GitHub、外部文档或方案对比
- reviewer：Design-only 阶段审查设计资产；Implementation-later 阶段审查实现变更
- verifier：Design-only 阶段核对验收清单和失败样例；Implementation-later 阶段跑验证和失败路径检查

然后按这个顺序输出：
1. 本阶段目标
2. 本阶段为什么重要
3. 本阶段防止哪类失控
4. 系统边界和责任分层
5. 最小可评审产物
6. 失败场景和架构评审问题
7. 验证方法或清单验收
8. 我应该如何复述
9. 是否进入下一阶段的判断
```

## 2.1 RAG 专项训练线

RAG 不再只作为 Phase 6 的一个小点，而是贯穿 Phase 3、5、6、7、8、9、11 的专项训练线。

| RAG 阶段 | 学什么 | 对应主线 |
|---|---|---|
| RAG-01 Baseline | PDF/Markdown/知识库最小问答 | Phase 3/5 |
| RAG-02 Chunking | normal、semantic、late chunking | Phase 6 |
| RAG-03 Query Rewrite | HyDE、multi-query、decomposition | Phase 6 |
| RAG-04 Hybrid + Rerank | BM25 + vector + cross-encoder/ColBERT | Phase 6/7 |
| RAG-05 Graph / Hierarchy | RAPTOR、GraphRAG、LightRAG | Phase 6/7 |
| RAG-06 Corrective | Self-RAG、CRAG、repair loop | Phase 4/7 |
| RAG-07 Eval | Ragas、Promptfoo、Giskard、AutoRAG | Phase 7/8 |
| RAG-08 Security | prompt injection、poisoning、tenant escape | Phase 9 |
| RAG-09 Cache + Cost | semantic cache、routing、topN 控制 | Phase 8/11 |
| RAG-10 Long-running Integration | index/version/checkpoint/context refs | Phase 4/8/11 |

学习原则：每个 RAG 优化都必须先复现失败，再做消融实验，最后用指标决定是否进入主架构。

2. `07-RAG问题诊断与优化/RAG闭环实验报告模板.md`
3. `22-评测集/RAG评测数据集设计.md`

## 4. 当前推荐 demo

首个 demo 建议使用：

**工单处理 Agent + 只读知识库 + 受控工具调用**

原因：

- 工单场景容易理解，适合教学。
- 只读知识库能降低安全风险。
- 受控工具调用可以覆盖网关、策略、审计、评估和记忆。
- 后续可以自然扩展到写操作审批、多 Agent 协作和长线任务恢复。

## 5. Phase 1 预习清单

进入 Phase 1 前，你只需要先理解四个词：

- **Agent manifest**：一个 Agent 的身份证，写清 owner、version、model、tools、policy、eval set。
- **ToolCall**：模型想调用工具时必须提交的结构化请求。
- **PolicyDecisionInput**：策略系统判断是否允许工具调用时需要看的事实。
- **ErrorModel**：所有错误必须可分类、可恢复、可审计。

Phase 1 不追求“Agent 很聪明”，只追求“Agent 做任何事都有结构、有边界、有证据”。

## 6. 2026-06-29 路线审核后的变化

本项目从 2026-06-29 起采用新版教学顺序：

- Phase 2 从“Gateway”改为“最小运行时”，先让你理解 Agent loop。
- Gateway 与 Tool Gateway 移到 Phase 3，并补上 Model Gateway、Policy Plane、Audit Plane。
- Runtime 的 checkpoint、resume、human-in-the-loop 拆到 Phase 4，专门讲长线任务。
- Evaluation 扩展为“测评审核与红队”，不仅测最终答案，也测工具、轨迹、记忆、安全、多 Agent。
- Observability 移到 Eval 之后，用 trace 和 replay 为评测与审计提供证据。
- 多 Agent 移到单 Agent 可治理之后，避免一开始就把复杂度放大。

## 7. 阶段完成后的复述句式

当一个阶段完成后，你应该能用下面句式复述：

```text
这一阶段解决了 [问题]。
我们新增/修改了 [产物]。
它参考了 [项目/协议/模式]，但没有照搬 [风险点]。
我用 [清单/样例/复述评分/测试] 证明它可以工作。
它仍然有 [剩余风险]，所以下一阶段要处理 [下一问题]。
```

如果这段话说不清楚，就不要急着进入下一阶段。
