# 第一节课 - 用工单故事理解工业级 Agent

生成日期：2026-07-01  
用途：在进入 Phase 1 之前，用一个具体业务故事把 Runtime、Gateway、Tool、Memory、Eval、Trace、Sandbox、Multi-Agent、Release 串起来。

## 1. 先看一个真实业务故事

公司有一个客服工单：

```text
客户说：我上个月买的企业套餐无法使用，请帮我退款。
```

一个普通聊天机器人可能会回答：

```text
好的，我建议给客户退款。
```

但工业级 Agent 不能这样草率。它如果真的要参与企业流程，至少要回答：

- 这个客户是谁？
- 他属于哪个租户？
- 他有没有退款资格？
- 退款政策是哪一版？
- 需要查哪些系统？
- 能不能直接退款，还是只能生成建议？
- 如果要退款，谁审批？
- 如果任务中断，怎么恢复？
- 如果 Agent 答错，怎么复盘？
- 如果发布了新 prompt 后策略变了，怎么回滚？

这就是工业级 Agent 和聊天 demo 的区别。

## 2. 把整个系统想成一条受控流水线

```text
用户请求
-> Request Gateway
-> Agent Runtime
-> Model Gateway
-> Tool Gateway
-> Policy
-> RAG / Memory / Tools
-> Trace / Audit
-> Eval / Release Gate
```

白话解释：

| 层 | 像什么 | 防什么事故 |
|---|---|---|
| Request Gateway | 公司前台 | 不知道谁来了、属于哪个租户 |
| Agent Runtime | 任务调度本 | 做到一半断了、步骤丢了 |
| Model Gateway | 模型调度员 | 模型乱选、成本失控 |
| Tool Gateway | 工具仓库管理员 | Agent 直接拿危险工具 |
| Policy | 权限审查员 | 低权限用户做高风险动作 |
| RAG | 资料室 | 回答没有证据、政策版本错 |
| Memory | 有权限的长期笔记 | 记错事实、越权召回 |
| Trace | 调试录像 | 出错后不知道哪一步错 |
| Audit | 审计记录 | 事后无法证明谁做了什么 |
| Eval | 考试系统 | 新版本退化却不知道 |
| Sandbox | 隔离执行间 | 高风险工具伤到真实环境 |
| Release Gate | 发布审批门 | 没测完就上线 |

## 3. 为什么不能直接让模型决定

坏设计：

```text
系统 prompt 写：你不能越权退款。
然后让模型自己决定是否调用退款工具。
```

问题是：

- 用户可以诱导模型忽略规则。
- 模型可能误解政策。
- prompt 不是权限系统。
- 事后没有结构化证据。
- 重试时可能重复退款。

正确方向：

```text
模型可以提出申请；
Tool Gateway 校验工具；
Policy 判断权限；
高风险操作进入审批；
Runtime 记录状态；
Audit 记录决定；
Eval 验证是否退化。
```

这一句话非常重要：

```text
Agent 的建议不等于系统的执行许可。
```

## 4. 这节课先不背术语

你先记 6 句话：

1. Agent 不是聊天窗口，而是会做多步任务的系统。
2. Runtime 管“任务怎么跑、怎么恢复”。
3. Gateway 管“谁能进来、能调用什么”。
4. Policy 管“这件事能不能做”。
5. Eval 和 Trace 管“怎么证明做对、错了怎么查”。
6. Release Gate 管“什么时候能上线、错了怎么回滚”。

## 5. 这条工单经过 11 个阶段会变成什么

| Phase | 在工单故事里做什么 |
|---|---|
| Phase 0 | 确认课程路线和学习方式 |
| Phase 1 | 定义工单、工具调用、权限判断、错误的结构 |
| Phase 2 | 定义工单处理任务的 run、step、state |
| Phase 3 | 让请求和工具调用都经过 Gateway/Policy/Audit |
| Phase 4 | 工单处理到一半中断后能恢复 |
| Phase 5 | 把知识库、CRM、工单系统作为受控工具接入 |
| Phase 6 | 只把可靠、授权、可删除的信息写入记忆 |
| Phase 7 | 设计评测，证明不会乱退款、乱引用、乱记忆 |
| Phase 8 | 用 trace/audit 定位每次失败 |
| Phase 9 | 高风险工具进入沙箱和审批 |
| Phase 10 | Planner、Executor、Reviewer、Verifier 分工协作 |
| Phase 11 | 带证据包发布、灰度、回滚、冻结 |

你不需要今天记住所有细节，只要知道它们都服务于同一个问题：

```text
让 Agent 从“会说”变成“能被企业安全地使用”。
```

## 6. 第一节课的设计卡

| 项目 | 内容 |
|---|---|
| 本课解决什么 | 区分聊天 demo 和工业级 Agent |
| 最小业务场景 | 工单处理 Agent + 只读知识库 + 受控工具调用 |
| 关键风险 | 越权、误操作、任务中断、无审计、无评测 |
| 必须存在的层 | Runtime、Gateway、Policy、Tool、Memory、Eval、Trace、Release |
| 当前不做什么 | 不写代码、不选框架、不接真实系统 |
| 过关证据 | 能用工单故事讲出每一层为什么存在 |

## 7. 坏设计卡

| 坏设计 | 为什么不行 |
|---|---|
| 只写一个强 prompt | prompt 不是权限边界 |
| Agent 直接调用退款工具 | 工具缺少策略、审批和审计 |
| 只保存聊天记录 | 长线任务无法恢复 |
| 记住所有客户信息 | 隐私、污染、越权风险 |
| 只测最终回答 | 看不见工具误选和权限绕过 |
| 没有发布门禁 | 新 prompt、新模型、新工具会悄悄改变行为 |

## 9. 三个追问

你可以回答这三个问题，回答不清楚就不要进入 Phase 1：

1. 为什么“建议退款”和“执行退款”必须分开？
2. 如果工单处理到一半系统崩溃，只保存聊天记录有什么问题？
3. 如果新版本 Agent 退款判断变差，怎么提前发现？
