# 非代码化工业级 Agent 设计学习协议

生成日期：2026-06-30  
用途：明确本课程可以采用 `Design-only` 学习模式。你不需要写代码或记代码，但必须学会设计、评审和验收工业级长线 Agent。

## 1. 核心定位

本项目不是让你背 API，也不是让你成为某个框架的搬运者。当前学习目标是：

```text
你能通过和 Codex 交互，设计出一个可恢复、可审计、可评测、可治理、可隔离、可回滚的工业级 Agent 系统。
```

`Design-only` 不等于降低标准。它只是把亲手实现代码延后，把当前重点放在：

- 你能不能问出正确问题。
- 你能不能识别错误设计。
- 你能不能判断方案是否达到工业级门槛。
- 你能不能把 Agent 的回答转化为架构图、契约、评测集、风险清单和上线门禁。

## 2. 两种学习模式

| 模式 | 当前是否推荐 | 你做什么 | Codex 做什么 | 验收看什么 |
|---|---|---|---|---|
| Design-only | 推荐 | 提问、追问、判断、复述、确认风险 | 解释、设计、生成文档、模拟评审、给验收标准 | 你是否能讲清设计、风险和门禁 |
| Implementation-later | 后续可选 | 选择要落地的模块和边界 | 写代码、跑测试、修复、验证 | 工程测试、评测集、trace、审计和发布门禁 |

当前默认采用 `Design-only`。进入代码阶段前，必须先确认：

- 你已经理解该阶段为什么存在。
- 你能指出至少一个坏设计。
- 你能说清最小验收标准。
- 你知道哪些证据能证明设计合格。

## 3. 每轮交互协议

每次学习或推进项目时，你可以直接使用：

```text
继续 Phase X 或 RAG-XX。

我采用 Design-only 模式，不写代码，也不记代码。
请先调用合适的 subagent 增强提示词；
如果涉及最新资料、GitHub、官方文档或方案对比，请调用 researcher；
然后按架构师教学方式输出：
1. 本轮要解决的失控风险；
2. 系统边界和责任分层；
3. 必须设计的契约或门禁；
4. 典型失败场景；
5. 架构评审问题；
6. 验收清单；
7. 我应该如何复述；
8. 是否能进入下一步。
```

## 4. 每阶段四个非代码化产物

| 产物 | 作用 | 示例 |
|---|---|---|
| 设计决策卡 | 说明为什么这样设计 | 为什么 Tool Gateway 必须拦截所有工具调用 |
| 失败场景卡 | 证明你知道系统怎么失控 | prompt injection 绕过知识库权限 |
| 架构评审问题 | 用来挑战 Codex 的方案 | 如果 Runtime 崩溃，checkpoint 如何恢复 |
| 用户复述验收 | 检查你是否真正理解 | 我能讲清状态机不是聊天记录 |

如果一个阶段只产生代码或工具名，但没有这四个产物，说明它不适合你的当前学习目标。

## 5. 你必须能识别的坏设计

| 坏设计 | 为什么危险 | 正确方向 |
|---|---|---|
| 用 prompt 代替权限 | 模型可被诱导，权限不可审计 | 权限在 Policy/Gateway 层做结构化判断 |
| 用 chat history 代替状态 | 长线任务无法恢复、重试和审计 | Run/Step/Checkpoint 显式建模 |
| 只评最终答案 | 工具选择、权限绕过、记忆污染都看不见 | 评估答案、轨迹、工具、记忆和安全 |
| Agent 直接拿工具凭据 | 泄漏和越权风险极高 | Credential proxy + Tool Gateway |
| 记忆默认写入 | 错误事实和敏感数据会长期污染 | Memory write gate + TTL + audit |
| RAG 只看向量相似度 | 引用、权限、新鲜度和多租户都可能错 | Retrieval + Policy + Citation + Eval |
| 多 Agent 自由聊天 | 任务冲突、重复执行、责任不清 | Handoff schema + resource lock + arbiter |

## 6. 阶段通过标准

每个阶段结束前，你不需要证明自己会写代码，但必须能回答：

- 这一阶段防止哪类 Agent 失控？
- 哪一层负责拦截、记录或恢复？
- 如果这层缺失，会出现什么生产事故？
- Codex 给出的方案有没有偷懒，比如把治理问题交给 prompt？
- 最小验收证据是什么？
- 哪些失败必须阻塞下一阶段？

## 7. 停止条件

出现下面情况时，不进入下一阶段：

- 你无法用自己的话复述本阶段边界。
- 方案中安全、权限、记忆、评测或审计责任不清。
- 外部项目或官方资料没有重新核验，却被当成“最新事实”使用。
- 评测只覆盖正常路径，没有失败路径、越权路径或回归路径。
- Codex 的回答只给技术名词，没有解释为什么、何时用、何时不用。

## 8. 最终目标

学完后，你应该能像架构负责人一样和 Agent 协作：

```text
我不需要亲手记住每行代码。
但我知道系统应该分几层，每层防什么风险；
我知道怎么要求 Codex 设计、实现、验证和复盘；
我知道如何判断一个方案能不能用于真实长线任务。
```
