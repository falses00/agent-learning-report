# 岗位能力与毕业标准

## 1. 先说结论

“看完课程”不能等于满足大厂招聘要求。只有学习者能独立实现、验证、解释并复盘一个完整系统，才接近 Agent/LLM 应用工程岗位的项目要求；学历、算法、通用后端、业务经历和面试仍是独立变量。

本课程的承诺边界是：提供覆盖岗位核心能力的训练路线和作品集门禁，不承诺 offer、职级或薪资。

## 2. 能力矩阵

| 能力域 | 合格证据 | 面试常见追问 |
|---|---|---|
| Python/Backend | 类型、异常、async、API、SQL、测试、Docker | 超时与重试如何设计？事务边界在哪？ |
| LLM 基础 | token/context、sampling、structured output、tool use | 为什么温度调低仍不等于确定性？ |
| Prompt/Context | versioned prompt、context budget、ablation | 哪段上下文真正提升了指标？ |
| Agent Runtime | state machine、step/event、cancel、resume | 聊天记录为什么不是任务状态？ |
| Tool/Policy | registry、schema、policy、approval、idempotency | 模型越权和重复副作用如何阻止？ |
| RAG | ingestion、ACL、hybrid/rerank、citation、eval | 召回率和答案正确率如何分开测？ |
| Memory | write gate、TTL、delete、tenant isolation | 错误记忆如何撤销并防止再召回？ |
| Eval | golden/trajectory/tool/security、judge calibration | 平均分高但关键 case 失败能发布吗？ |
| Observability | traces、metrics、audit、SLO、cost | 如何定位一次高延迟和成本暴涨？ |
| Security | threat model、prompt injection、secret、sandbox、MCP | 间接 prompt injection 为什么不是过滤关键词？ |
| Reliability | checkpoint、retry taxonomy、ledger、rollback | 工具成功但响应丢失时如何恢复？ |
| Multi-agent | handoff、budget、loop guard、baseline comparison | 为什么多 Agent 不一定更好？ |
| Delivery | CI/CD、migration、canary、runbook、incident review | 一个模型/prompt 更新如何安全发布？ |
| Communication | ADR、architecture doc、demo、postmortem | 你做过的最大取舍是什么，证据呢？ |

## 3. 毕业作品集必须包含

1. 可运行仓库：一条命令启动，一条命令测试，一条命令跑 eval。
2. 架构说明：系统上下文、容器/模块、关键时序、信任边界。
3. 12 个以上 ADR：记录关键决策、备选方案、撤销条件。
4. 不少于 60 个自动化测试，其中至少 20 个失败或对抗用例。
5. 不少于 100 个版本化 eval cases，覆盖答案、RAG、工具、轨迹和安全。
6. 一份 baseline 与优化版对比报告，包含质量、P95 延迟、成本和关键失败率。
7. 三份事故复盘：重复副作用、跨租户/注入风险、模型或检索退化。
8. 一次 crash/resume 演示和一次 rollback drill。
9. 一份 threat model 和一套可运行 adversarial suite。
10. 5-10 分钟演示视频或脚本：正常路径、受阻路径、恢复路径各一条。

数量不是质量的替代物；所有证据都必须与 OpsPilot 的真实风险相关。

## 4. 最终答辩场景

毕业答辩必须现场完成：

- 解释请求从 Gateway 到 Runtime、Tool、Policy、Storage、Eval 的完整链路。
- 改坏一个 policy 或 prompt，展示 CI/eval 如何阻塞。
- 模拟工具执行成功但网络断开，证明恢复不会重复退款。
- 注入跨租户检索请求，证明数据不泄漏并产生 audit event。
- 从 trace 定位一次慢请求，给出证据驱动的优化方案。
- 解释为什么当前多 Agent 设计优于单 Agent；若无证据，应退回单 Agent。

## 5. 评分标准

总分 100，80 分毕业，且安全、可靠性、评测三项都不得低于各自及格线。

| 维度 | 分值 | 及格线 |
|---|---:|---:|
| 通用工程基础 | 15 | 9 |
| Agent 架构与 Runtime | 15 | 10 |
| Tool/Policy/Reliability | 15 | 10 |
| RAG/Memory/Context | 15 | 9 |
| Eval/Red Team | 15 | 10 |
| Security/Privacy | 10 | 7 |
| Observability/SRE/Release | 10 | 6 |
| 表达与作品集 | 5 | 3 |

直接判不毕业：存在跨租户泄漏；高风险写操作可绕过审批；重复执行有资金/数据副作用；关键安全 eval 失败仍可发布；无法复现自己的结果。

## 6. 求职材料转化

简历 bullet 应写“问题 + 设计 + 量化结果 + 约束”，例如：

```text
Built a durable ticket-resolution agent with policy-gated tools and idempotent
resume; prevented duplicate side effects across crash/retry tests and added
trajectory/security eval gates for every prompt, tool, and retrieval release.
```

不要写“熟悉 LangChain/会 Prompt Engineering”这类无法验证的标签。面试材料应能打开具体代码、测试、trace 和复盘报告。

## 7. 自查清单

- 我能不用框架术语解释每一层解决的事故。
- 我能从零写最小 Runtime，而不是只会调用 Agent API。
- 我能证明工具权限来自系统事实，不来自 prompt。
- 我能区分单元测试、离线 eval、在线指标和人工审核。
- 我能给出模型、RAG、工具、状态、安全各自的失败归因路径。
- 我能解释系统仍不具备哪些生产条件。
- 我能现场复现、修复并回归一个失败。

任一项只能回答“文档里写了”，说明还没学会。
