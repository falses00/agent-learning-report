# 教学构建循环

生成日期：2026-06-28  
适用项目：Agent Runtime Gateway  
目标：让用户不仅拿到一个 Agent 项目，还能一步一步理解、复现、评估、改造和定制自己的企业级 Agent。

## 1. 总原则

每一阶段都必须同时完成三件事：

1. **产出设计或工程结果**：设计卡、契约、评测集、文档、代码、测试、评估或报告。
2. **教学说明**：解释为什么做、怎么做、怎么知道做对了。
3. **反思决策**：判断资料是否最新、方案是否先进、是否应该继续下一阶段。

当前默认模式是 `Design-only`：用户不写代码，也不记代码。先用交互方式掌握设计、评审和验收；代码实现留给后续 `Implementation-later`。

默认循环：

```text
Survey
-> Subagent Prompt Enhancement
-> Freshness Reflection
-> Advanced Preflight
-> Plan
-> Design / Execute
-> Teach
-> Verify / Acceptance Review
-> Review
-> Continue/Stop Decision
-> Reflect
```

Design-only 循环：

```text
Survey
-> Subagent Prompt Enhancement
-> Explain Boundary
-> Challenge Bad Design
-> Design Review
-> Failure Drill
-> Acceptance Checklist
-> User Restatement
-> Reflect
```

## 2. 每轮启动协议

每次用户发起任务后，先生成增强提示词：

```text
用户原始需求：
[原文]

增强后的任务：
目标：
输入：
输出：
边界：
学习模式：Design-only / Implementation-later
相关文件：
需要调用的 subagent：
需要搜索/核验的最新资料：
执行步骤：
验收标准：
暂停条件：
```

规则：

- 搜索资料、当前信息、GitHub 项目、官方文档时，优先调用 `researcher` subagent。
- 复杂阶段启动前调用 `planner` subagent。
- 非平凡修改后调用 `reviewer` 或 `verifier`。
- subagent 默认只读，主线程负责最终判断。

## 3. 阶段模板

```text
阶段 N：{阶段名称}

A. 目标
- 本阶段解决什么问题？
- 它服务于 PRD 的哪一部分？
- 完成后用户应该理解什么？

B. 输入
- 本地文件：
- 外部资料：
- 上一阶段输出：
- 待确认问题：

C. Subagent 分工
- planner：
- researcher：
- explorer：
- reviewer：
- verifier：

D. 实施步骤
1. Survey：读取本地上下文。
2. Freshness：核验最新资料。
3. Plan：拆分输出。
4. Design：先形成可评审的设计卡、失败场景和验收清单。
5. Execute：如果进入 Implementation-later，才做本阶段范围的代码或配置。
6. Teach：写教学说明。
7. Verify：跑检查或对照清单。
8. Review：独立审查。
9. Reflect：决定继续、暂停或修正。

E. 输出
- 更新文件：
- 新增文档：
- 设计决策卡：
- 失败场景卡：
- 架构评审问题：
- 用户复述验收：
- 验证命令或清单：
- 教学说明：
- 下一阶段输入：

F. 继续条件
- 本阶段验收全部通过。
- 关键结论有来源或假设标记。
- 用户能理解并复述本阶段目的。
- 下一阶段输入明确。

G. 暂停条件
- 关键资料无法核验。
- 产品/架构方向需要用户选择。
- 安全或合规边界不清。
- 三轮修复后仍无法通过验证。
```

## 4. 教学说明模板

每阶段结束后必须给用户一段教学说明：

```text
本阶段教学说明：{阶段名称}

1. 这一阶段在解决什么？
用 3-5 句话解释本阶段的产品或架构意义。

2. 为什么必须先做这一步？
说明它对后续 PRD、架构、代码、测试或评估的影响。

3. 我具体做了哪些动作？
- 查了哪些资料：
- 读了哪些本地文件：
- 做了哪些判断：
- 写入/修改了哪些产物：

4. 你如何知道这一步实施到位？
- 有明确输入和输出。
- 有来源链接或假设标记。
- 能回答为什么不是另一种方案。
- 能映射到 PRD 或架构章节。
- 有验收清单或验证命令。

5. 你自己如何复现？
1. 打开相关文件。
2. 查看证据矩阵。
3. 对照阶段目标。
4. 检查关键结论是否有来源。
5. 用验收清单逐项打勾。

6. 常见错误
- 只列名词，不解释用户价值。
- 只列 GitHub 项目，不说明采用/不采用理由。
- 把 LLM Gateway、Agent Runtime、Tool Gateway 混成一层。
- 只写 prompt 规则，不落到契约、policy、eval、trace 或后续可实现的测试。
- 把“不写代码”误解成“不需要验收”。
- 没有继续/暂停条件。

7. 下一阶段如何承接？
说明下一阶段会使用本阶段哪些输出。
```

## 5. Freshness Reflection

每阶段开始前必须问：

| 检查项 | 通过标准 |
|---|---|
| 是否查了最新资料 | 搜索/核验 GitHub 或官方文档，标注日期 |
| 是否存在版本变化 | release、README、维护状态、迁移说明已检查 |
| 是否区分来源类型 | 官方文档、GitHub README、代码、第三方文章、产品假设分开 |
| 是否有过期风险 | 有则写入风险或待核验 |
| 是否更新了本地文档 | 发现不能只留在聊天里，必须写入文件 |

## 6. Advanced Preflight / Reflection

每阶段开始执行前和结束前都必须问。开始前用于判断“是否值得做、是否足够先进”；结束前用于判断“是否真的完成、是否应该继续”：

- 是否只做了聊天能力，而没有做运行时能力？
- 是否覆盖了 checkpoint、resume、human-in-the-loop？
- 是否把工具调用放到了 Gateway 和 policy 后面？
- 是否有 schema、runtime validation、test？
- 是否考虑 tenant、identity、audit、secret？
- 是否考虑 eval、trace、cost、latency？
- 是否考虑 sandbox、egress、filesystem、network？
- 是否有清晰的“不能做什么”？

## 7. Continue / Stop Decision

继续下一阶段前必须满足：

- 本阶段产物已落盘。
- 本阶段教学说明已给出。
- 本阶段验收通过。
- 下一阶段目标清晰。
- 没有阻塞性用户决策。

暂停并询问用户的情况：

- MVP 场景不明确。
- 需要真实凭据或外部账号。
- 需要执行破坏性操作。
- 两个架构分支都合理但取舍不同。
- 安全、合规、部署边界无法假设。

## 8. 长线阶段路线

| 阶段 | 目标 | 核心产物 | 用户应该学会 |
|---|---|---|---|
| 0 | 课程总览与路线审核 | PRD、research、traceability、route review | Agent Runtime Gateway 是什么，路线为什么这样排 |
| 1 | 契约层 | ToolCall schema、Agent manifest | 为什么 schema 是边界 |
| 2 | 最小运行时 | Run/Step、mock model、mock tool、trace_id | Agent loop 如何运行 |
| 3 | Agent 网关与工具治理 | Request/Model/Tool/Policy Gateway、Audit | Prompt 不能决定权限 |
| 4 | 长线任务与断点恢复 | Checkpoint、resume、retry、HITL | 长任务如何恢复 |
| 5 | MCP 与工具生态 | MCP adapter、tool registry、credential broker | MCP 解决什么、不解决什么 |
| 6 | 记忆系统 | memory store、write gate、retrieval policy、memory eval | 记忆如何可信进入上下文 |
| 7 | 测评审核与红队 | golden/redteam/trajectory eval/CI gate | 如何证明准确率和安全性 |
| 8 | 可观测性与审计 | trace/cost/replay/audit export | 如何复盘失败 |
| 9 | 安全隔离与沙箱 | Docker/gVisor/E2B/microVM profile | 不可信执行如何隔离 |
| 10 | 多智能体协同 | handoff、shared state、resource lock、arbiter | 协作如何不失控 |
| 11 | 治理控制台与发布 | policy、approval、audit、release gate | 企业如何上线 |

## 9. Phase 1 教学式执行 Playbook

阶段目标：

- 设计 Agent manifest、ToolCall、PolicyDecisionInput、ErrorModel 四个核心契约。
- 让你理解“Agent 的第一道安全边界不是 prompt，而是运行时 schema”。

### 9.1 跟做练习

1. 画出一个 Agent manifest：
   - `agent_id`
   - `version`
   - `owner`
   - `allowed_tools`
   - `memory_scope`
   - `risk_level`
   - `eval_set`
   - `budget`

2. 画出一个 ToolCall：
   - `tool_name`
   - `tool_version`
   - `arguments`
   - `principal`
   - `operation_id`
   - `reason`
   - `dry_run`
   - `approval_ticket`

3. 设计 5 个坏输入：
   - 缺少 `operation_id`
   - 额外字段 `admin_override`
   - `tool_name` 不在 allowlist
   - `reason` 中包含 prompt injection
   - 写操作却没有 `dry_run=false` 和审批票据

4. 设计 validation 判定表：
   - 正常输入通过。
   - 5 个坏输入必须失败。
   - 失败错误必须结构化。
   - 后续 Implementation-later 再把判定表转成自动化测试。

### 9.2 如何知道 Phase 1 做对

- 任何模型输出都不能直接进入工具层。
- 每个工具调用候选都能被 schema 接住。
- 用户能解释每个字段为什么存在。
- Design-only 阶段能用坏输入验收表证明哪些输入应被拒绝。
- Implementation-later 阶段再用测试证明坏输入被真实 schema 拒绝。

### 9.3 常见错误

- 把 ToolCall 写成 `dict[str, Any]`。
- 只在 prompt 里要求格式正确，没有 runtime validation。
- schema 没有 `extra=forbid`。
- 忘记 `operation_id`、`principal`、`reason`、`dry_run`。
- 错误响应不可机器解析。

### 9.4 Phase 1 进入条件

- MVP demo 场景已写入需求文档或场景选择文档，并能复述其边界。
- PRD、research、traceability 已通过审查。
- 已确认 Phase 1 只做契约层，不提前实现 Runtime 或 Gateway。

## 10. 每阶段验收问卷

用户可以用下面问题检查自己是否真的理解：

1. 我能用一句话说清楚本阶段目标吗？
2. 我知道本阶段输入和输出分别是什么吗？
3. 我知道哪些内容来自官方资料，哪些是我们的产品假设吗？
4. 我知道为什么要参考这些 GitHub 项目，又为什么不能照搬吗？
5. 我知道本阶段对应哪些 PRD 需求吗？
6. 我能说出至少 3 个失败路径吗？
7. 我知道如何验证这一步是否做对吗？
8. 我知道下一阶段为什么依赖当前阶段吗？

## 11. 当前阶段教学说明

阶段：Phase 0 - 课程总览与路线审核

这一阶段在解决什么：

我们先定义产品，而不是先写代码。因为企业级 Agent 的难点不是让模型回答问题，而是把模型的工具调用、长期状态、记忆、评估、安全和审计放进可靠系统。PRD 是后面架构、代码、测试和评估的契约。

为什么必须先做：

如果没有 PRD，后续很容易变成“看到哪个框架火就接哪个框架”。这样会做出一个堆概念的 demo，而不是企业可用的 Runtime Gateway。PRD 先规定边界：什么是 MVP，什么是 P1/P2，什么必须做，什么坚决不做。

如何知道做对了：

- PRD 能解释产品定位。
- GitHub research 能说明参考项目和取舍。
- traceability 能把需求映射到来源和验收。
- 每个核心模块都有验收标准。
- 后续代码阶段可以直接从需求 ID 开始实现。

下一阶段承接：

Phase 1 Design-only 会从 PRD 中的 `Agent manifest`、`ToolCall schema`、`PolicyDecisionInput`、`ErrorModel` 开始，把“契约边界”落成契约设计、字段责任、坏输入样例和验收清单；进入 Implementation-later 后，再把这些设计落成代码和测试。最小运行时留到 Phase 2，`Tool Gateway` 留到 Phase 3。
