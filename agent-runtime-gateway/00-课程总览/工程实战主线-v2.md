# 工程实战主线 v2

版本：2026-07-12

定位：以作品集和生产思维为目标的唯一主线，替代“先学完 11 个 Design-only 阶段，再决定是否实现”的旧顺序。

## 1. 最终交付

学习者要独立交付 **OpsPilot 企业工单 Agent**：支持知识问答、工单分类、退款建议、人工审批、受控执行、长任务恢复、评测、审计和发布回滚。

毕业演示不是“模型回答了一句话”，而是一次可解释的系统演练：

```text
用户请求 -> 身份/租户校验 -> Runtime -> 检索/规划 -> ToolCall
-> Policy -> Approval -> Tool Gateway -> State/Checkpoint
-> Answer/Citation -> Trace/Audit -> Eval -> Release Gate
```

## 2. 推荐节奏

标准节奏 20 周，每周 10-15 小时。已有后端经验可压缩到 12-14 周；基础薄弱则先增加 4-8 周基础训练。不要按日历硬过关，只按证据过关。

| 阶段 | 周期 | 核心问题 | 必做工程增量 | 必做故障注入 | 通过证据 |
|---|---:|---|---|---|---|
| F0 工程基础 | 0-4 周 | 能否独立写、测、调后端程序 | Python、HTTP、SQL、Git、Docker 小实验 | 超时、坏 JSON、事务回滚、测试失败 | pytest、API 调用、SQL 记录、Git 历史 |
| S0 基线与契约 | 1 周 | 模型输出如何变成可信输入 | contracts、mock planner、CLI、schema tests | 缺字段、未知工具、跨租户 | validation tests、结构化错误 |
| S1 Runtime | 1-2 周 | Agent 如何可控地一步步运行 | Run/Step/Event、状态机、max steps、cancel | 模型超时、循环、非法状态跳转 | 状态转换测试、trace |
| S2 Tool/Policy Gateway | 2 周 | 模型为何不能直接执行工具 | registry、policy、approval、audit、idempotency | prompt 越权、重复退款、参数注入 | allow/deny/approval 测试、audit |
| S3 RAG | 2 周 | 如何得到有来源且不越权的答案 | ingestion、hybrid retrieval、rerank、citation | 找不到、找错租户、过期文档、注入文档 | retrieval/answer eval、citation report |
| S4 Durable Execution | 2 周 | 中断后如何安全恢复 | checkpoint、resume、retry policy、outbox/ledger | 进程崩溃、审批超时、工具成功但响应丢失 | crash/replay test、无重复副作用 |
| S5 Memory/Context | 1 周 | 什么该记，什么必须忘 | session summary、long-term write gate、TTL、delete | 错误事实、PII、跨租户召回 | memory eval、删除/过期证明 |
| S6 Eval/Red Team | 2 周 | 如何证明版本没有退化 | golden、trajectory、tool、security eval runner | 平均分掩盖 critical failure | 分层指标、blocking cases、CI report |
| S7 Observability/SRE | 1-2 周 | 事故时如何定位、止损和复盘 | OTel traces、metrics、logs、audit、SLO | provider 慢、成本暴涨、trace 缺失 | dashboard/trace、告警、复盘报告 |
| S8 Security/Sandbox | 2 周 | 不可信输入和工具如何隔离 | threat model、secret broker、egress/FS policy、sandbox | indirect injection、exfiltration、SSRF、恶意 MCP | adversarial suite、阻断证据 |
| S9 Multi-agent | 1 周 | 何时拆分真的值得 | planner/reviewer 或 specialist handoff | 循环委派、冲突写入、错误多数票 | 与单 Agent 的质量/成本/延迟对比 |
| S10 Release/Portfolio | 2 周 | 如何上线、回滚并证明能力 | container、migration、canary、release packet、runbook | 坏 prompt、坏索引、坏迁移 | rollback drill、演示视频、架构答辩 |

## 3. 每阶段固定实验结构

每个阶段新增一个 `labs/SXX-*` 实验包，必须包含：

```text
README.md          # 问题、边界、步骤、预期证据
failure.md         # 如何复现故障及根因
src/ or patch      # 最小工程增量
tests/             # 正常、失败、对抗检查
evals/             # 适用时加入回归 case
evidence/README.md # 命令、关键输出、trace/eval 链接
adr.md             # 决策、备选方案、取舍、撤销条件
```

禁止只提交截图。证据必须能从代码和命令重新生成。

## 4. 分阶段教学内容

### F0：工程基础

必须会：Python 类型与异常、async 基础、HTTP、JSON、SQL 事务、Git、pytest、Docker、Linux/PowerShell 基础、环境变量和日志。算法基础至少覆盖数组/哈希/栈队列/树图/复杂度；系统设计基础至少理解缓存、队列、幂等、一致性、限流和可观测性。

常见问题：只会 Notebook；不会定位栈追踪；把重试当成万能修复；把 secret 写进仓库。解决：用 CLI/API 项目训练，任何失败先写最小复现和测试。

### S0：基线与契约

实现：`AgentManifest`、`RunRequest`、`ToolCall`、`PolicyDecision`、`ToolResult`、`ErrorModel`。先使用 deterministic planner，避免一开始把错误归因给模型随机性。

常见问题：schema 太松、业务错误只有字符串、信任模型生成的 tenant/role。解决：边界校验、错误分类、身份事实由网关注入。

### S1：Runtime

实现：显式状态机、step limit、timeout、cancel、event log。把“聊天历史”和“任务状态”分开。

常见问题：一个 `while True` 包办所有逻辑；失败后不知道从哪一步恢复；隐藏状态塞在 prompt。解决：状态枚举、合法转换表、结构化 Step/Event、每步输入输出可重放。

### S2：Tool/Policy Gateway

实现：Tool Registry、参数校验、策略判定、人工审批、credential boundary、operation ledger。模型只申请，网关才执行。

常见问题：prompt 里写“不要越权”就当鉴权；重试导致重复扣款；工具返回内容再次注入模型。解决：事实驱动 policy、idempotency key、tool output 视为不可信数据。

### S3：RAG

实现：文档解析、chunk/version、tenant ACL、hybrid retrieval、rerank、引用、拒答、离线评测。先做 baseline，再用实验决定是否引入 HyDE、GraphRAG 等复杂方法。

常见问题：只调 `top_k`；只看答案主观感觉；检索命中却引用错误；索引更新不可追溯。解决：拆分 retrieval/answer/citation 指标，版本化 corpus 和 index，critical ACL case 阻塞发布。

### S4：Durable Execution

实现：checkpoint、resume token、retry classification、idempotency、outbox/operation ledger、human-in-the-loop timeout。

常见问题：把 checkpoint 当聊天记录；恢复时重复写操作；不区分 recoverable/terminal。解决：保存业务状态和已提交副作用，用 operation id 查询既有结果，按错误类别恢复。

### S5：Memory 与 Context Engineering

实现：短期工作上下文、摘要、长期记忆写入门禁、来源、TTL、撤销、租户隔离、上下文预算。

常见问题：所有对话都长期保存；把模型总结当事实；删除后向量库仍可召回。解决：最小化存储、来源与置信度、同步删除、memory eval 和 context ablation。

### S6：Eval 与 Red Team

实现：dataset schema、golden/tool/trajectory/RAG/security eval、LLM-as-judge 校准、回归门禁。先用确定性断言，主观质量才用 judge，并保留人工校准集。

常见问题：只看平均分；测试数据污染；judge 自洽偏差；线上事故不回流。解决：关键 case 单独阻塞、数据版本化、多人/规则校准、事故自动进入 regression set。

### S7：Observability 与 SRE

实现：trace/span/metric/log/audit 分工，token/cost/latency/error 指标，SLO、错误预算、告警、replay 和事故复盘。

常见问题：日志很多但无法按 run 串联；trace 记录 secret；只有 provider 延迟，没有端到端延迟。解决：统一 correlation ids、字段脱敏、分层 span、采样与保留策略。

### S8：Security 与 Sandbox

实现：资产/信任边界、least privilege、secret broker、network egress、filesystem policy、sandbox profile、MCP server 准入和 supply-chain 检查。

常见问题：把 prompt injection 只当文本过滤；Docker 等于绝对隔离；工具拥有长期生产凭据。解决：隔离不可信内容与指令、短期凭据、域名/方法 allowlist、强沙箱和人工审批。

### S9：Multi-agent

先测单 Agent baseline。只有角色专业化、上下文隔离或独立验证带来可量化收益时才拆。实现显式 handoff、预算、最大深度、资源所有权和终止条件。

常见问题：角色越多越“智能”；按投票合并事实；循环委派；共享上下文失控。解决：最少角色、按证据合并、handoff schema、loop guard、独立 reviewer/verifier。

### S10：Release 与作品集

实现：配置和迁移、容器、CI、staging、canary、版本锁定、release gate、rollback、runbook、SLO 复盘。演示必须主动展示一次失败和恢复。

常见问题：只部署代码，不锁 prompt/model/tool/index/eval；没有回滚演练；作品集只有架构图。解决：release manifest、自动门禁、定期故障演练、保留可复现实验报告。

## 5. 框架学习策略

课程核心先用小型自研 Runtime 理解边界，再选择一个主框架完成对比迁移：

| 选择 | 适合 | 课程要求 |
|---|---|---|
| OpenAI Agents SDK | OpenAI 模型、tools、handoffs、tracing 的快速工程 | 仍需自己实现业务 policy、数据治理、eval 和发布门禁 |
| LangGraph | 状态图、durable execution、HITL、复杂工作流 | 必须解释 checkpoint、idempotency 和 state schema |
| Google ADK | Google Cloud/多模型生态、agent/team 工程 | 不把框架内对象直接当企业治理边界 |
| 自研最小 Runtime | 学习核心机制、避免黑盒 | 只做教学和关键业务差异，不造完整通用框架 |

禁止同时主修三个框架。主线只选一个，另两个用于同一用例的短对比实验。

## 6. 阶段门禁

每阶段按 0-3 分评分：设计解释、实现质量、测试深度、评测证据、故障处理、工程表达。任一项 0 分或总分低于 14/18 不得继续。

- 0：没有证据或结论错误。
- 1：能跑 happy path，但无法解释/恢复失败。
- 2：正常与失败路径完整，边界基本清楚。
- 3：有对抗验证、量化证据、取舍与复盘，能回答追问。

最终完整标准见[岗位能力与毕业标准](岗位能力与毕业标准.md)。
