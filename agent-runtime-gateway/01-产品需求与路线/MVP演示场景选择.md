# MVP演示场景选择

生成日期：2026-06-28  
状态：建议采用，等待用户最终确认  
用途：在进入 Phase 1 前，先确定第一个可教学、可验证、风险可控的 Agent demo。

## 1. 选择原则

首个 demo 不追求炫技，追求覆盖企业级 Agent 的关键问题：

- 有真实业务目标。
- 有工具调用，但不能默认允许高风险写操作。
- 能覆盖权限、审计、记忆、评估和长线任务。
- 能设计坏输入和越权测试。
- 能逐步扩展到更复杂场景。

## 2. 候选场景对比

评分范围：1 分最低，5 分最高。

| 场景 | 教学清晰度 | 企业贴近度 | 安全可控 | 覆盖模块 | 实施复杂度 | 总评 |
|---|---:|---:|---:|---:|---:|---|
| 工单处理 Agent | 5 | 5 | 4 | 5 | 3 | 首选 |
| 企业知识库 Agent | 5 | 4 | 5 | 3 | 2 | 适合做子模块 |
| 代码执行/修复 Agent | 4 | 4 | 2 | 5 | 5 | 后期加入 |
| 浏览器自动化 Agent | 4 | 4 | 2 | 4 | 5 | 后期加入 |

说明：实施复杂度分数越高，表示越复杂。

## 3. 推荐结论

首个 MVP demo 使用：

**工单处理 Agent + 只读知识库 + 受控工具调用**

最小故事：

```text
用户提交一个工单问题。
Agent 读取工单内容和知识库片段。
Agent 判断工单类型、风险等级和建议处理动作。
Agent 可以调用只读工具查询上下文。
如果需要写操作，只生成待审批动作，不直接执行。
所有请求、工具调用、策略判断、结果都进入审计日志。
```

## 4. 为什么不是先做代码执行 Agent

代码执行 Agent 很有价值，但不适合作为第一阶段 demo：

- 沙箱要求更高。
- 文件系统、shell、网络和依赖安装都带来额外风险。
- 教学时容易把注意力从 Agent 治理转移到代码执行细节。
- 没有网关、策略、审计和评估前，代码执行会放大安全问题。

代码执行 Agent 建议放到 Phase 9 安全隔离之后，作为高风险工具样例加入。

## 5. MVP-slice 验收故事

Phase 3 结束时，最小演示必须能跑通：

1. 用户发起工单处理请求。
2. Request Gateway 生成 `request_id`、`thread_id`、`trace_id`。
3. Agent 产生结构化 `ToolCall`。
4. Tool Gateway 对参数做 schema 校验。
5. Policy 判断只读查询允许、写操作拒绝或转人工审批。
6. Mock 工具返回结果。
7. Audit Event 记录请求、主体、工具、策略决策和结果。

## 6. MVP-complete 验收故事

Phase 9 结束时，应能证明：

- 工单处理任务可以中断后从 checkpoint 恢复。
- 同一个 `operation_id` 重试不会重复产生副作用。
- 只读知识库召回不会越租户。
- 长期记忆写入必须经过 Memory Write Gate。
- golden dataset 和 red team dataset 可以阻止质量退化。
- trace 能展示一次请求经过的模型、工具、策略、记忆和评估步骤。
- 高风险工具必须进入 sandbox 或 human-in-the-loop。

## 7. 后续扩展路线

| 扩展方向 | 进入阶段 | 作用 |
|---|---|---|
| 写操作审批 | Phase 4 | 教会 human-in-the-loop 和恢复 |
| MCP 工具接入 | Phase 5 | 替换 mock tool，接入标准协议 |
| 长期用户偏好 | Phase 6 | 教会记忆写入和召回治理 |
| 工单准确率评估 | Phase 7 | 教会 golden dataset 和回归门禁 |
| 可观测 dashboard | Phase 8 | 教会 trace 和故障定位 |
| 代码执行工具 | Phase 9 | 教会高风险沙箱 |
| 多 Agent 协同处理 | Phase 10 | 教会 supervisor/worker/reviewer 分工 |
