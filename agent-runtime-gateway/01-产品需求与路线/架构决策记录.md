## 3. ADR-001：首个 MVP demo 场景

状态：提议中  
日期：2026-06-28

背景：需要一个既适合教学，又能覆盖企业级 Agent 治理能力的第一个 demo。

候选方案：

- 工单处理 Agent。
- 企业知识库 Agent。
- 代码执行/修复 Agent。
- 浏览器自动化 Agent。

决策：推荐使用“工单处理 Agent + 只读知识库 + 受控工具调用”。

原因：

- 比纯知识库更能覆盖工具、策略、审计和长线任务。
- 比代码执行和浏览器自动化更安全，适合作为第一条教学主线。
- 后续可以自然扩展到审批、写操作、MCP、记忆和多 Agent 协同。

需要复核的条件：

- 用户明确更想优先做代码执行或浏览器自动化。
- 后续目标转为开源框架而不是企业内部平台。

## 4. ADR-002：开源项目采用方式

状态：已接受  
日期：2026-06-28

决策：所有参考项目分三类处理：

- `dependency allowed`：可作为候选依赖继续评估。
- `adapter candidate after license review`：许可证/商业条款复核后才可作为适配器候选。
- `pattern only`：只学习设计模式，不直接引入依赖。

原因：GitHub star、活跃度和知名度不能等价于企业生产可引入。许可证、维护状态、安全边界和架构耦合都必须单独评估。

## 5. ADR-003：文档命名方式

状态：已接受  
日期：2026-06-28

决策：保留 `README.md` 作为默认入口，目录和教学文档使用中文命名。

原因：

- `README.md` 是 GitHub 和很多工具的默认入口。
- 中文文件名更适合教学阅读。
- 代码包名、API 路径和命令暂不中文化，避免后续工具链兼容问题。

## 6. ADR-004：MCP 采用边界

状态：提议中  
日期：2026-06-28

决策：MCP 作为工具、资源、提示词的标准化协议层；企业级权限、凭据、审计、限流、结果过滤由 Tool Gateway 补齐。

原因：MCP 解决“怎么接工具”，不等于解决“谁能用工具、何时能用、如何审计、如何隔离”。

## 7. ADR-005：MVP-slice 与 MVP-complete 区分

状态：已接受  
日期：2026-06-28  
更新：2026-06-29

决策：

- `MVP-slice`：Phase 3 结束，证明 Runtime + Gateway + Tool + Policy + Audit 链路可行。
- `MVP-complete`：Phase 9 结束，runtime、gateway、checkpoint、MCP、memory、eval、observability、sandbox 基本打通。

原因：如果把所有 P0 都塞进 Phase 2，会导致教学节奏和工程范围失控。

## 8. ADR-006：教学路线顺序

状态：已接受  
日期：2026-06-29

决策：后续教学采用：

```text
Phase 1 契约层
-> Phase 2 最小运行时
-> Phase 3 Agent 网关与工具治理
-> Phase 4 长线任务与断点恢复
```

原因：

- 先做契约层，可以把 Agent 行为、工具调用、策略事实和错误结构化。
- 再做最小运行时，学习者能理解 Agent loop、Run、Step、Trace。
- 再做 Gateway，学习者能理解为什么要把运行时放进企业治理边界。
- 长线任务恢复单独成章，避免和 Gateway 混在一起导致验收不清。

## 9. ADR-007：测评审核体系独立成章

状态：已接受  
日期：2026-06-29

决策：测评审核不再只放在单一 Evaluation 阶段里，而是作为贯穿全项目的体系，并在 Phase 7 集中成章。

测评对象包括：

- 最终答案。
- 工具调用。
- Agent 轨迹。
- 记忆系统。
- Gateway。
- 安全治理。
- 多 Agent 协同。
- 可观测性和审计。

原因：企业级 Agent 的准确率不是单一分数。长线任务、记忆污染、越权、重复工具调用、多 Agent handoff 失败都可能让系统不可用。

## 10. ADR-008：工业级框架标准

状态：已接受  
日期：2026-06-29

决策：本项目即使处在教学和 MVP-practical 阶段，也必须按工业级 Agent Framework 的骨架推进。

不可妥协标准：

- Run/Step/State/Event 显式建模。
- Checkpoint/Resume/Retry/Cancel 设计。
- Tool Gateway 统一工具校验、授权、审计。
- Memory 写入门禁和召回权限。
- Golden/Red Team/Trajectory/Memory Eval。
- Trace/Audit 覆盖模型、工具、策略、记忆和审批。
- Release Gate 和 Rollback。

原因：如果前期为了快而绕过这些骨架，后期从 demo 变成实际应用时会推倒重来。

## 11. ADR-009：多租户字段策略

状态：已接受  
日期：2026-06-30

决策：Phase 1 的 schema 从 Day 1 保留 `tenant_id`、`workspace_id`、`principal_id`、`data_classification`、`policy_scope` 等字段，但 MVP-practical 运行模式可以先按单租户执行。

原因：

- RAG、Memory、Tool Gateway、Audit 和 Policy 都会被多租户边界影响。
- 如果 schema 不预留多租户字段，Phase 6/9 再补会导致大量返工。
- 单租户运行能降低第一版复杂度，但不能删除边界字段。

影响：

- Phase 1 契约层必须包含 tenant/workspace/principal 字段。
- Phase 2-4 可以用默认租户运行。
- Phase 6 RAG/Memory 必须开始测试跨租户隔离。
- Phase 9 之前必须完成多租户安全红队。
