## 1. 一句话定位

Agent Runtime Gateway 是一个企业级 Agent 基础设施层，用于把不稳定的 LLM/Agent 行为放入可恢复、可审计、可评估、可隔离、可治理的工程边界中。

它不是单个聊天机器人，也不是单纯 LLM Gateway，更不是只靠多角色提示词拼出来的 Agent demo。它要统一管理：

- 用户请求进入 Agent 的入口。
- Agent 的注册、版本、运行、暂停、恢复、回滚、审计。
- 模型路由、成本、限流、fallback。
- MCP/工具接入、权限、凭据、参数校验、结果过滤。
- 长线任务状态机、checkpoint、human-in-the-loop。
- 记忆系统、检索、遗忘、权限隔离。
- 自动评估、红队测试、回归门禁。
- 沙箱、安全隔离、越权拦截、合规审计。

## 3. 背景与问题定义

企业要把 Agent 放进真实业务系统时，会遇到这些问题：

1. **长线任务不可控**  
   Agent 运行几十步后中断、工具失败、上下文爆炸、人工插队，状态无法恢复。

2. **工具调用高风险**  
   模型可能把“查询订单”变成“修改订单”，或者把用户输入当作工具参数直接执行。

3. **权限边界不清**  
   Prompt 里的“你不能越权”不是安全机制。企业需要真正的 RBAC/ABAC、租户隔离、工具白名单、审批和审计。

4. **记忆系统容易污染**  
   错误事实、过期偏好、跨租户数据、敏感信息可能被写入长期记忆，并在未来任务中被错误召回。

5. **质量无法证明**  
   改模型、改 prompt、加工具后，如果没有 golden dataset、red team dataset、trace 和 CI gate，就无法知道准确率是否退化。

6. **沙箱与外部系统隔离不足**  
   让 Agent 执行代码、浏览器、shell、数据库工具时，普通进程或普通容器不足以承担高风险隔离要求。

7. **多 Agent 协作容易失控**  
   角色对话式 Agent 容易循环、互相等待、重复调用工具或把错误状态传播给其他 Agent。

## 4. 产品愿景

长期愿景：成为企业内部所有 Agent 进入业务系统前的统一运行时与治理入口。

成功后的形态：

- 平台团队可以注册 Agent、工具、模型、策略和评估集。
- Agent 开发者可以用任意框架构建 Agent，但必须通过统一 Runtime Gateway 运行。
- 安全团队可以审计每次工具调用、权限决策、记忆写入、模型调用和人工审批。
- 业务团队可以将 Agent 应用接入工单、知识库、CRM、数据库、浏览器、代码执行等系统，同时保留回滚和人工接管能力。
- 运维团队可以查看成本、延迟、失败原因、沙箱资源、任务恢复率和回归评估结果。

## 6. 目标与非目标

### 6.1 MVP 目标

MVP 要证明：Agent 可以在受控网关里完成一个长线任务，并且每一步都能校验、追踪、恢复和评估。

MVP 必须包括：

- Agent Registry：注册 Agent manifest、版本、owner、权限范围。
- Request Gateway：统一入口、身份、trace_id、thread_id。
- Tool Gateway：schema 校验、RBAC、operation_id、audit log。
- MCP-compatible Tool Adapter：支持最小 Tools/Resources/Prompts 映射。
- Runtime State Machine：任务 state、step、checkpoint、resume。
- Human-in-the-loop：高风险工具调用暂停审批。
- Memory v1：session memory + artifact summary，不做默认 GraphRAG。
- Evaluation v1：golden/redteam 数据集和基础 CI gate。
- Observability v1：模型调用、工具调用、checkpoint、成本和错误 trace。
- Sandbox v1：禁网络、限时、限资源的 Docker/gVisor 可选执行策略。

### 6.2 P1 目标

- 多 Agent 协作：Router、Planner、Executor、Reviewer、Verifier。
- MCP Server Registry：内部 curated registry + 外部 registry mirror。
- Memory v2：vector retrieval、metadata filter、重要性/时效性打分。
- Eval v2：Promptfoo/Ragas/Langfuse/Phoenix 后端适配。
- Policy v2：ABAC、审批票据、租户级策略继承。
- Sandbox v2：E2B/Firecracker/microVM provider 抽象。
- Console：Agent、Run、Tool、Trace、Eval、Policy 管理界面。

### 6.3 P2 目标

- GraphRAG/causal memory。
- 多模型智能路由与成本优化。
- 多工作区并发执行和资源锁。
- 私有 MCP registry。
- 企业级 SSO/SCIM/SAML/OIDC 集成。
- 自动生成评估报告和合规报表。
- 多云/混合部署。

### 6.4 非目标

- 不做通用聊天机器人。
- 不替代所有业务系统，只作为 Agent 运行与治理入口。
- MVP 不做完整低代码工作流平台。
- MVP 不默认执行任意不可信代码。
- MVP 不默认启用 GraphRAG。
- 不把某个开源框架绑定为唯一运行时；框架通过 adapter 接入。

### 7.1 必须作为设计基准的能力

| 能力 | 参考项目 | PRD 采用方式 |

| 编码/工具型 Agent 安全 | OpenHands、Codex、Cline | 借鉴 workspace、diff、审批、终端、MCP、文件权限组合模型 |

| 沙箱隔离 | gVisor、Firecracker、E2B、Wasmtime | 分级执行环境，不把 Docker 当成唯一安全边界 |

### 7.2 不照搬的内容

- 不照搬 CrewAI/AutoGen 的黑盒多角色对话控制流。
- 不照搬 AutoGPT 的早期自主循环。
- 不把 Dify/OpenHands/Cline 当作通用 Runtime Gateway 模板，它们更多是应用平台或具体 Agent 产品。
- 不直接使用公共 MCP Registry 作为企业内部可信 registry；企业需要 curated/private registry。
- 不把 Langfuse/Phoenix 绑定成唯一观测后端；底层事件应兼容 OpenTelemetry。

## 8. 长线阶段性任务循环与教学式交付

本项目的构建不是一次性写完代码，而是长期循环。每一阶段都必须交付两类结果：

1. 设计或工程产物：设计卡、契约、评测集、文档、代码、测试、评估、报告。
2. 教学说明：解释为什么做、怎么做、如何判断做对、下一阶段怎么承接。

当前产品学习目标不是让用户亲自记住代码，而是让用户在与 Codex 的交互中掌握工业级 Agent 的设计、评审和验收能力。工程实现是后续可选承接，不能替代当前的设计理解。

### 8.0 Design-only 与 Implementation-later

| 模式 | 用途 | 用户责任 | Codex 责任 | 通过标准 |
|---|---|---|---|---|
| Design-only | 当前默认模式 | 提问、追问、复述、判断风险 | 解释设计、生成文档、模拟评审、给验收清单 | 用户能说明边界、失败、门禁和下一步 |
| Implementation-later | 后续工程模式 | 选择落地范围和验收门槛 | 写代码、跑测试、修复、验证 | 工程测试、评测集、trace、audit、release gate 通过 |

无论采用哪种模式，工业级门槛不降低：Gateway、Runtime、Tool Gateway、Policy、Memory、Eval、Observability、Sandbox、Governance 都必须有边界和验收。

每完成一个阶段，必须向用户说明：

### 8.4 Freshness / Continue / Stop 检查

Freshness 检查：

- 是否重新核验关键 GitHub/官方资料。
- 是否标注访问日期。
- 是否发现项目维护状态变化。
- 是否区分官方来源、GitHub README、第三方文章和产品假设。

Advanced 检查：

- 是否覆盖 Agent runtime，而不只是模型代理。
- 是否覆盖 checkpoint、resume、human-in-the-loop。
- 是否覆盖 MCP/tool protocol 生态。
- 是否覆盖 tenant、policy、identity、audit、secret。
- 是否覆盖 eval、observability、dataset、prompt version。
- 是否覆盖 sandbox、prompt injection、data exfiltration。

继续条件：

- 本阶段验收清单全部通过。
- 下一阶段输入明确。
- 没有必须由用户选择的重大分歧。
- 用户能通过教学说明复述本阶段目的和验收方式。

暂停条件：

- 关键资料无法核验。
- 架构方向出现两个以上合理分支。
- 安全/合规边界不清。
- 连续三轮修复仍无法通过验证。

## 10. 核心用户流程

### 10.1 Agent 创建与发布

1. 开发者创建 Agent manifest。
2. 选择模型、工具、记忆范围、运行策略。
3. 上传测试用例和评估集。
4. 平台校验 schema、权限、成本预算、安全策略。
5. reviewer 审核。
6. 发布为 draft/beta/production。
7. 后续每个版本都保留 diff、eval result、approval record。

验收：

- 没有 owner、权限范围、评估集的 Agent 不能进入 production。
- 生产 Agent 必须绑定 rollback 版本。

### 10.2 用户请求进入 Gateway

1. 用户通过 API/UI/Webhook/IDE 提交任务。
2. Request Gateway 生成 `request_id`、`thread_id`、`trace_id`。
3. AuthN/AuthZ 识别用户、租户、工作区。
4. Prompt Gateway 清洗输入并注入规则摘要。
5. Model Gateway 选择模型。
6. Runtime 创建 Run 和初始 checkpoint。
7. Agent 开始执行。

验收：

- 每个请求都有可追踪 ID。
- 未授权用户无法进入工具层。
- 超长输入、空输入、非法 JSON 有统一错误。

### 10.3 工具调用流程

1. Agent 产生工具调用意图。
2. Tool Gateway 校验 tool schema。
3. Policy Gateway 检查 principal、scope、tenant、risk。
4. 低风险读操作直接执行。
5. 高风险写操作进入 dry-run 或 human approval。
6. 执行前生成 `operation_id` 幂等键。
7. 执行结果经过 result schema 和 redaction。
8. 写 audit event 和 trace span。

验收：

- Prompt 永远不能决定权限。
- 所有工具调用必须有 schema、principal、operation_id、trace_id。
- 破坏性操作必须有审批或显式禁用策略。

### 10.4 长线任务暂停与恢复

1. Run 执行多个 Step。
2. 每个 Step 成功后写 checkpoint。
3. 发生 timeout、工具失败、高风险动作或用户中断时暂停。
4. 用户或系统选择 resume/retry/fork/cancel/escalate。
5. 继续执行时从 checkpoint 恢复，不重复已提交副作用。

验收：

- kill 进程后可从最近 checkpoint 恢复。
- 重复 `operation_id` 不产生重复副作用。
- 高风险 Step 可人工接管。

## 11. 核心模块需求

### 11.1 Agent Gateway

功能：

- 提供统一请求入口。
- 管理 authentication、authorization、tenant/workspace。
- 注入策略摘要、上下文、可用 Agent 列表。
- 路由到正确 Agent、模型和 runtime。
- 提供错误归一化。

输入：

- 用户请求、API key/JWT/session、tenant_id、workspace_id、agent_id 或 task_type。

输出：

- run_id、stream events、final result、error object。

异常场景：

- 身份失效、租户不存在、Agent 未发布、超出速率、预算不足。

验收：

- 所有入口请求有 trace。
- 未授权请求返回标准错误。
- 入口不直接执行工具。

### 11.2 Model Gateway

功能：

- OpenAI-compatible API adapter。
- 多模型路由、fallback、重试预算。
- 成本估算、token 计量、budget guard。
- 模型级策略：高风险任务禁止低能力模型自动执行。

参考：LiteLLM、Portkey。

验收：

- 每次模型调用记录 model、prompt version、token、cost、latency。
- fallback 不隐藏质量退化，必须写 trace。

### 11.3 Agent Runtime

功能：

- 管理 Run、Step、State、Checkpoint。
- 支持 durable execution、resume、retry、fork、cancel。
- 支持 human-in-the-loop。
- 支持 loop guard 和 deadlock guard。

参考：LangGraph、Temporal。

验收：

- 状态不是 chat history，而是结构化 state。
- 每个 Step 可重试、可观察、可终止。
- checkpoint 恢复测试通过。

### 11.4 Orchestration

功能：

- 单 Agent 流程。
- 多 Agent 协作：Router、Planner、Executor、Reviewer、Verifier。
- DAG/StateGraph/事件驱动。
- 子任务拆分、任务锁、文件/资源所有权。

验收：

- 多 Agent 不能直接互相传未校验自由文本作为工具参数。
- 重复状态、重复边、最大步数有拦截。
- 资源锁使用确定顺序，避免死锁。

### 11.5 MCP / Tool Gateway

功能：

- MCP server registry。
- MCP Tools/Resources/Prompts 发现与治理。
- Tool schema 管理、参数校验、结果过滤。
- 凭据注入、tool-level permission、rate limit。
- tool version、server health、熔断、审计。

参考：MCP specification、MCP Registry、MCP SDK、GitHub MCP Server、FastMCP、Composio。

验收：

- 公共 MCP server 必须经过 curated allowlist。
- 私有 MCP server 不依赖公共 registry。
- Tool Gateway 补齐 MCP 不提供的企业权限、审计和租户治理。

### 11.6 Memory System

功能：

- Working memory：当前 Run state。
- Session memory：当前 thread 的短期上下文。
- Semantic memory：租户/项目知识检索。
- Tool result cache：可复用工具结果。
- Optional graph memory：实体/关系/事件。

参考：LlamaIndex、GraphRAG、RedisVL。

验收：

- 长期记忆写入必须有来源、时间、置信度、可删除路径。
- 记忆召回必须受 tenant/workspace/tool policy 约束。
- 不保存 secret、token、一次性敏感信息。
- 记忆污染有纠错和失效机制。

### 11.6.1 RAG / Retrieval System

功能：

- 文档解析、切块、索引、检索、重排、上下文打包、引用。
- 支持 baseline RAG、hybrid search、query rewrite、rerank、GraphRAG/LightRAG 候选、Self/Corrective RAG 候选。
- 支持多租户 ACL、metadata filter、source priority、freshness、conflict mode。
- 支持 retrieval trace、index version、embedding version、reranker version、context pack version。

验收：

- 任何 RAG 优化必须有 baseline、失败样例、指标对比和回归集。
- 检索前必须做权限过滤，不能只在生成后过滤。
- RAG answer 必须能追溯 citation、retrieved_doc_ids、index_version、policy_decision。
- GraphRAG、LightRAG、ColBERT、CRAG、semantic cache 等高级方案必须先通过消融实验，不能因为热门就默认启用。

### 11.7 Evaluation System

功能：

- Golden dataset。
- Red team dataset。
- Tool call accuracy。
- RAG faithfulness/context recall。
- Long-run recovery tests。
- Security block tests。
- CI eval gate。

参考：Promptfoo、Ragas、Langfuse、Phoenix、OpenAI evals/Inspect AI 类思路。

验收：

- prompt/model/tool/agent manifest 变更必须触发回归。
- 破坏性工具未拦截直接 fail。
- 指标不能只依赖 LLM-as-judge，关键安全用例必须规则断言。

### 11.8 Observability

功能：

- Trace/span/log/metric。
- 每个 Run 的 model call、tool call、memory hit、policy decision、checkpoint。
- 成本、延迟、失败率、重试、审批。
- OpenTelemetry exporter。

参考：OpenTelemetry、Langfuse、Phoenix。

验收：

- 任一失败能定位到 prompt/model/memory/tool/policy/sandbox 哪一层。
- 任一工具调用能回答：谁、何时、为何、参数、结果、审批、重试。

### 11.9 Security & Isolation

功能：

- Tenant isolation。
- User/Agent/Tool identity。
- Secret vault。
- RBAC/ABAC/ReBAC。
- Prompt injection 防护。
- DLP/redaction。
- Sandbox provider。
- Egress control。
- Approval policy。

参考：OPA、Kong、Envoy、Casdoor、gVisor、Firecracker、E2B、Wasmtime。

验收：

- 模型看不到原始 secret。
- 未授权工具无法被 prompt 绕过。
- 不可信代码默认禁网络、限资源、隔离文件系统。
- 安全事件写入不可抵赖审计日志。

### 11.10 Governance Console

功能：

- Agent registry。
- Tool registry。
- Policy editor。
- Run explorer。
- Trace/eval dashboard。
- Cost dashboard。
- Approval inbox。
- Audit export。

验收：

- 管理员可禁用 Agent、工具、MCP server。
- 安全团队可搜索某用户/Agent/工具的行为链。
- 运维能按 tenant/workspace 看到成本和失败趋势。

### 14.3 策略类型

- RBAC：角色允许哪些工具和资源。
- ABAC：根据租户、数据级别、时间、环境、风险动态判断。
- ReBAC：用户与资源关系，例如 owner/reviewer/approver。
- Approval policy：高风险操作需要人工审批。
- Budget policy：成本或 token 超限暂停。
- Egress policy：工具允许访问的网络目标。

## 17. 里程碑

本节是权威 Phase Map。`P0` 表示 MVP-complete 必须具备的能力，但不代表都在 Phase 1 完成。2026-06-29 路线审核后，本项目采用“契约层 -> 最小运行时 -> 网关治理”的教学顺序。

为了保持可教学、可验收、可演示，本项目把 MVP 拆成两个边界：

- **MVP-slice**：Phase 3 结束时形成最小端到端演示，证明 Runtime + Gateway + Tool + Policy + Audit 链路可行。
- **MVP-complete**：Phase 9 结束时 P0 能力基本打通，包含 runtime、gateway、checkpoint、MCP、memory、eval、observability、sandbox。

## 18. 高风险未决问题

1. MVP 是否定位为“企业内部平台”还是“开源开发者框架”？
2. 首个 demo 场景选择工单、代码执行、知识库还是浏览器自动化？
3. 是否允许接入真实外部系统，还是先用 mock tool？
4. 沙箱优先本地 Docker/gVisor，还是优先 E2B/云沙箱？
5. 评估后端优先 Promptfoo/Ragas，还是 Langfuse/Phoenix？
6. MCP server 是否只支持本地 stdio，还是 MVP 支持 remote HTTP？
7. 是否要求多租户从 Day 1 落地，还是先用单租户但保留 schema？
8. 许可证风险如何处理：NOASSERTION/ELv2/商业条款项目只能参考，不能嵌入。

## 19. MVP 范围收缩建议

为了真正交付，MVP 应主动砍掉：

- 完整 no-code workflow UI。
- 默认 GraphRAG。
- 完整私有 MCP registry。
- 多云部署。
- 复杂 ReBAC。
- 自动 Agent marketplace。
- 真实生产凭据接入。

MVP 只保留：

- 一个可运行的 Agent Runtime。
- 一个可治理的 Tool Gateway。
- 一个可恢复的长线任务。
- 一个可审计的工具调用。
- 一套可回归的评估数据。
