# 路线审核与反思-2026-06-29

审核日期：2026-06-29  
审核方式：`researcher` subagent + 主线程 GitHub CLI 核验 + 本地文档一致性检查  
范围：PRD、学习路线、需求追踪、GitHub 调研、教学手册、测评审核体系。

## 1. 本次审核结论

原路线已经覆盖企业级 Agent 的关键模块，但教学顺序需要调整：

- 原路线偏“企业治理先行”：契约层之后立刻进入 Gateway。
- 新路线改为“先理解 Agent 怎么跑，再进入企业治理”：契约层之后先做最小运行时，再做网关与工具治理。
- 记忆系统、评估审核、多 Agent 协同需要更细，不应只作为单个模块名称出现。
- GitHub 调研需要补充中小项目，尤其是记忆、协同、观测、评估、human-in-the-loop 和 MCP workflow。

最终建议路线：

```text
Phase 0  课程总览与路线审核
Phase 1  契约层
Phase 2  最小运行时
Phase 3  Agent 网关与工具治理
Phase 4  长线任务与断点恢复
Phase 5  MCP 与工具生态
Phase 6  记忆系统
Phase 7  测评审核与红队
Phase 8  可观测性与审计
Phase 9  安全隔离与沙箱
Phase 10 多智能体协同
Phase 11 治理控制台与发布
```

## 2. 第一轮反思：原路线是否能拿来实用

能作为企业级 Agent 基础设施路线使用，但还不够“可教学”。

优点：

- 覆盖 Runtime、Gateway、Tool、Memory、Eval、Observability、Sandbox、Governance。
- 已经把 MVP-slice 和 MVP-complete 分开。
- 已经要求 subagent、验证、审查和反思。

不足：

- Gateway 放在 Runtime 前面，适合平台工程师，但对学习者不够直观。
- 记忆系统只有“能记”和“能召回”，还缺写入门禁、冲突消解、压缩、遗忘和记忆评测。
- 多 Agent 协同不能只讲角色，需要讲 handoff 协议、共享状态、仲裁、锁、超时和回滚。
- 测评不能只测最终答案，需要测工具调用轨迹、成本、延迟、记忆污染、越权、安全拒答和回归稳定性。

## 3. 第二轮反思：GitHub 小中型项目补充后缺口是什么

新增调研显示，很多几千 star 项目在某个专项上很强，适合用来补课程细节：

- 记忆：Zep、Mem0、Letta、Graphiti、TencentDB-Agent-Memory、A-Mem、Hindsight。
- 协同：AG2、CAMEL、AgentVerse、Langroid、MindSearch、agentUniverse。
- 观测与评估：AgentOps、Agenta、Opik、LangWatch、Giskard、Promptfoo、Phoenix。
- 网关与工具治理：Portkey、Plano、mcp-agent、mcp-use。
- 审批与安全：HumanLayer、Invariant、Garak。

补充后路线应新增三个强约束：

1. **Memory Eval 不能缺席**：记忆系统完成标准必须包括命中率、污染率、跨租户泄漏率、过期清理率。
2. **Agent 轨迹必须可审计**：每个工具调用、模型调用、policy 决策、memory read/write 都要有 trace/span。
3. **多 Agent 必须有协议而不是聊天**：handoff、共享状态、任务锁、仲裁和停止条件必须结构化。

## 4. 第三轮反思：哪些项目不能照搬

不能因为 star 高或 README 好看就直接引入：

- Swarm 更适合 handoff 教学，不能作为生产路线。
- AgentVerse 适合学习多 Agent simulation，但维护节奏和重构状态需谨慎。
- HumanLayer 当前更适合理念参考，不能依赖仓库代码。
- TensorZero、TaskWeaver、Rebuff 等若处于 archived 或迁移状态，只能做历史模式参考。
- 托管优先的平台类项目要警惕数据驻留、遥测、许可证和商业条款。

本项目采用原则保持不变：

- `dependency allowed`：许可证清楚、边界清楚、可替换。
- `adapter candidate after license review`：有价值，但先审许可证、部署方式、商业条款。
- `pattern only`：只学习模式，不直接引入。

## 5. 第四轮反思：最优教学顺序

最优教学不是“按流行框架顺序学”，而是“按企业风险出现顺序学”。

推荐顺序：

1. 先让 Agent 的输入、工具调用、错误和策略事实结构化。
2. 再做一个最小可运行 Agent，让学习者知道 Agent loop 是什么。
3. 再把 Agent 放进 Gateway、Policy、Audit 边界里。
4. 再处理长线任务恢复、MCP 工具生态、记忆系统。
5. 再建立测评审核、可观测性、安全隔离。
6. 最后做多 Agent 协同和治理控制台。

这样学习者每一步都能回答：

```text
这一层解决哪个失控风险？
它怎么被验证？
它失败时从哪里复盘？
它是否能进入下一阶段？
```

## 6. 本次路线变更

| 旧阶段 | 新阶段 | 原因 |
|---|---|---|
| Phase 2 Gateway MVP Slice | Phase 2 最小运行时 | 先让学习者看到 Agent loop、Run、Step、Trace |
| Phase 3 Runtime + Checkpoint | Phase 3 Agent 网关与工具治理；Phase 4 长线任务 | 把网关治理和长线恢复拆开，更容易验收 |
| Phase 6 Evaluation | Phase 7 测评审核与红队 | 强化审核体系，不只做准确率 |
| Phase 7 Observability | Phase 8 可观测性与审计 | Eval 先定义要看什么，Observability 再提供证据 |
| Phase 8 Sandbox | Phase 9 安全隔离与沙箱 | 安全隔离基于前面的工具、观测和评测结果 |
| Phase 9 Multi-Agent | Phase 10 多智能体协同 | 多 Agent 必须建立在单 Agent 可治理之后 |
| Phase 10 Governance | Phase 11 治理控制台与发布 | 治理是最终上线闭环 |

## 7. 进入 Phase 1 前的最新判断

可以进入 Phase 1，但要先接受两个教学原则：

- Phase 1 只做契约层，不提前实现 Gateway。
- Phase 2 做最小运行时，不追求完整模型能力，用 mock model 和 mock tool 先证明 loop、trace、schema 和错误模型。

如果这两个原则成立，Phase 1 的目标就很清楚：把 `AgentManifest`、`ToolCall`、`PolicyDecisionInput`、`ErrorModel` 做成后续所有阶段共用的地基。
